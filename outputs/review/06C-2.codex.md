### 지표는 통과하지만 목적을 배신하는 경로
- 어떻게 통과시키는가: `knownExceptions`에 실제 failure subject를 모두 넣고, 동시에 임의의 extra subject를 추가한다. 무엇이 배신되는가: record가 “정확히 알려진 legacy 오염”이 아니라 사람이 확인하지 않은 예외 목록을 신뢰 기록으로 굳힌다. 지금 구현에 실재하는가: 예, `server/TrustOriginCli.cs:154-158`은 extra를 검사하지 않는다.

- 어떻게 통과시키는가: dirty worktree에서 `WORKSTATE.json`과 applier-log만 clean하게 유지하고 declare를 실행한다. 무엇이 배신되는가: `baselineCommit`은 HEAD인데 실제 검증/실행 소스는 dirty tree일 수 있어 “clean checkout A”가 깨진다. 지금 구현에 실재하는가: 예, `CheckNoDirectMod`는 두 handoff 파일만 본다(`server/TrustOriginCli.cs:456-460`).

- 어떻게 통과시키는가: 자동 launcher 활성 여부나 high-risk fail-closed 검증을 하지 않아도 production 기본값으로 통과한다. 무엇이 배신되는가: bootstrap command가 “강한 안전장치”를 확인했다는 record를 만든다. 지금 구현에 실재하는가: 예, `server/TrustOriginCli.cs:183-188`.

### 테스트가 못 잡는 것
- extra known exception: self-test는 unlisted failure만 본다. 입력: `knownExceptions=[{"subject":"LISTED-TRANSITION",...},{"subject":"FAKE-EXCEPTION",...}]`, WORKSTATE는 `LISTED-TRANSITION`만 failure가 나게 구성. 기대는 거부 또는 extra 보고지만 현재는 통과한다.

- production git/precondition 경로: self-test는 `TestPrecon`으로 `BuildVerified/AutoLauncherOff/HighRiskClosed`를 주입한다(`server/TrustOriginCli.cs:285`, `:315`, `:364`, `:381`, `:395`). 실제 git 상태, 실제 launcher 상태, 실제 high-risk command fail-closed를 반증하지 않는다.

- record schema completeness: self-test는 `trustEpoch`, `baselineCommit`, `declarationCommit` 부재, `reconciliationVerdict`만 확인한다. `phaseChangeReady:false`, `replayReady:false` 필드 누락은 검사하지 않는다.
