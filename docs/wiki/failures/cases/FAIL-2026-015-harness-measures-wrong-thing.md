# FAIL-2026-015 — 하네스가 **다른 것을 잰다**: 초록불이 대상과 무관하다

- 상태: 진행 중 (06C-1-R2 · 05H-R3 · CODEX-GATE-04에서 개별 대응)
- 최초 발생일: 2026-07-12 (검수자가 "게이트가 다섯 번 거짓말했다"로 처음 묶음)
- 최근 발생일: 2026-07-14
- 관련 DI: 05H · 05H-R2 · 06C-1 · 06C-1-R1 · GUARD-01 · CODEX-GATE-04
- 구성요소: verification-harness
- failureClass: verification_gap
- 심각도: 높음 (게이트가 **거짓 초록**을 준다 — 모든 판정의 신뢰가 무너진다)

## 발생 상황

이 저장소는 게이트(하네스)의 exit code로 판정한다. **그런데 게이트가 "무엇을" 재는지 아무도 검사하지 않는다.**
2026-07-12~14 사이 **같은 형태의 사고가 아홉 번** 나왔다. 전부 **게이트는 초록인데 대상은 빨간** 경우다.

초록불을 믿었으면 **손 위조 transition-id · rollback 부재 · `--human-decision` 위조 구멍**이 그대로 land됐을 것이다.

## 발생 이유

- **직접 원인**: 각 사례마다 다르다(인코딩·구성·집합 연산·확장자 목록·접두사 매칭·인자 파싱).
- **근본 원인 (공통)**: **하네스에 대한 반증 시험이 없다.**
  하네스는 "대상이 잘못됐을 때 빨간불을 켜는가"를 **자기 자신에 대해 증명하지 않는다.**
  **판정하는 자를 아무도 판정하지 않는다.**
- **기여 원인**:
  - 완료 기준이 **exit code**로만 적혀 있다. "무엇을 재는가"는 산문이라 기계가 검사 못 한다.
  - 실행자가 지표(exit 0)를 맞추도록 압박받는다 — **범위를 줄이거나 규칙을 완화하면 초록이 된다.**
  - **지시서와 프롬프트가 모순되면** 실행자가 완화 쪽으로 밀린다(→ FAIL-2026-016).

## 검토한 해결 방법

1. **하네스마다 자기 반증 시험 필수화** — "고치기 전엔 뚫렸다"를 보이지 않으면 반려.
2. **`--self-test`(in-process 단언 실행기) 패턴** — 고정 fixture + **하드코딩 기대값**, 인자 미개방.
3. 게이트 매니페스트에 "무엇을 재는가"를 기계 판정 가능한 형태로 등재 (`CLI-CONTRACT.json`·`GATE-MANIFEST.json`).
4. 하네스가 자기 배설물을 세지 않게 byproduct ignore set (scope-check).

## 선택한 해결 방법

**1 + 2를 규칙으로 승격한다.**

- **모든 하네스 지시서의 완료 기준에 "고치기 전 재현" 항목을 넣는다.** 없으면 반려.
- **`--self-test` 패턴을 표준으로 한다.** 05H-R2가 `handoff-integrity --self-test`로 처음 만들었고,
  06C-1-R2가 `state-transition --self-test`로 따른다.
  - in-process 단언 실행기 · 고정 fixture · **기대값 하드코딩** · **경로/id/플래그 인자 미개방**(열면 production 우회로).
  - **실패의 *원인*(코드 집합)까지 단언한다.** "어쨌든 실패했으니 통과"는 금지(사례 5).
  - **`HarnessError`(입력이 깨짐)와 `Failure`(검사가 잡음)를 구분한다.**
- 3·4는 `CODEX-GATE-04`에서 별도 처리.

## 판단 기준

**"계측기는 자기가 틀렸을 때 그것을 말할 수 있어야 한다."**
초록을 주는 이유가 **"기대한 그 이유로 통과했기 때문"**이어야 한다.

## 검증 결과

- 05H-R1: 규칙 2 복원 후 검수자 반증 시험 F 재실행 → **05H exit 0(거짓 PASS) → 05H-R1 exit 1** (고쳐짐)
- 05H-R2: `--self-test` 신설. **단, 실패 원인 미단언 → 05H-R3로 반려**
- 06C-1-R1: `.txt`·접두사·silent ignore → **06C-1-R2로 반려**
- **아홉 사례 중 다섯이 아직 미해결**(05H-R3 · 06C-1-R2 · CODEX-GATE-04 대기)

## 재발 방지

1. **지시서 작성 시**: 완료 기준에 **"★ 고치기 전 재현"** 항목을 반드시 넣는다.
2. **하네스 제작 시**: `--self-test`를 함께 만든다. **하네스만 만들고 자기 시험을 안 만들면 미완성이다.**
3. **검수 시**: 하네스가 초록을 주면 **하네스를 먼저 반증한다.** 대상을 보기 전에 계측기를 의심한다.
4. **`legacyCallsiteCount=0` 같은 "0"은 증거가 아니다.** **0을 만들 수 있는 다른 방법**(범위 축소·확장자 누락·접두사 매칭)이 있는지 먼저 본다.

## 후속 작업과 잔여 위험

- `05H-R3` — `--self-test`가 실패 코드를 단언하게 (사례 5)
- `06C-1-R2` — silent ignore · `.txt` 범위 · 접두사 allowlist · `state-transition --self-test` (사례 7·8·9)
- `CODEX-GATE-04` — Debug 바이너리 · CLI 계약 · scope-check 잡음 · claim liveness PID 프록시 (사례 1·2·3)
- **잔여 위험**: 이 목록 자체가 **완전하지 않다.** 아홉 건은 **찾은 것**이고, 찾지 않은 게이트가 더 있다.
  **`GATE-MANIFEST`에 등재되지 않은 검사는 아무도 돌리지 않는다.**
