# FAIL-2026-003 — 승인된 화면에 새 제안이 떠 승인 버튼이 막힌 것처럼 보임

- 상태: 해결됨
- 최초 발생일: 2026-07-10
- 최근 발생일: 2026-07-10
- 관련 DI: 커밋 4e8c39c 계열 승인 UI 수정
- 구성요소: dashboard-approval, proposal-rendering
- failureClass: ui_state, known_failure
- 심각도: high

## 발생 상황

proposal이 승인된 뒤 새 제안이 렌더링되며 승인 버튼 상태가 어긋났다. 사용자는 새로 보이는 화면에서 승인 버튼을 눌렀지만, "눌러도 반응 없음"처럼 보였다. 별개로 백그라운드 탭이 최신 상태를 못 받아오던 `visibilitychange` 미갱신 문제도 있었다.

## 관찰된 증상과 영향

사람이 승인을 못 하는 것으로 오인했다. 실제로는 승인 상태와 새 proposal 렌더링 상태가 엇갈려 UI가 현재 결재 가능 여부를 명확히 표현하지 못했다.

## 발생 이유(직접·근본·기여, 미확정은 가설 표시)

- 직접 원인: 승인 후 proposal lifecycle과 렌더링 상태가 어긋나 버튼이 기대한 상태로 동작하지 않는 것처럼 보였다.
- 근본 원인: 미확정. 승인 직후 새 제안이 표시되는 상태 전환과 버튼 활성 조건의 정확한 경계는 추가 관찰이 필요하다.
- 기여 요인: 백그라운드 탭이 `visibilitychange` 시 최신 상태를 가져오지 않아 오래된 상태가 화면에 남을 수 있었다.
- 가설: `lifecycle === "submitted"` 처리와 승인 완료 후 새 proposal 표시 타이밍이 함께 작용했을 가능성이 있다.

## 검토한 해결 방법

- 승인 버튼 클릭 로직만 보강한다.
- proposal lifecycle이 `submitted`일 때만 결재 가능한 상태로 렌더링한다.
- 탭이 다시 보일 때 최신 상태를 다시 가져온다.

## 선택한 해결 방법

커밋 `4e8c39c` 계열에서 상태 렌더링과 `lifecycle === "submitted"` 처리를 보강하고, `visibilitychange` 시 최신 상태를 갱신하도록 했다.

## 판단 기준

- 승인된 proposal과 새 proposal을 UI에서 혼동하지 않아야 한다.
- 결재 가능한 상태가 아니면 버튼이 막힌 이유가 상태와 일치해야 한다.
- 백그라운드에서 돌아온 탭은 최신 서버 상태를 가져와야 한다.

## 검증 결과

해결은 커밋 `4e8c39c` 계열로 반영된 것으로 기록한다. 이 사례의 근본 원인 일부는 미확정으로 남기며, 재발 시 동일 증상이 상태 전환 문제인지 최신 상태 동기화 문제인지 추가 조사한다.

## 재발 방지

- proposal 결재 버튼은 `lifecycle === "submitted"` 상태와 연결한다.
- `visibilitychange` 시 최신 상태를 다시 불러온다.
- 승인 완료 화면과 새 proposal 화면의 상태 전환을 후속 관찰 대상으로 둔다.

## 후속 작업과 잔여 위험

- 근본 원인 일부가 미확정이다.
- 재발 시 승인 직후 proposal ID, lifecycle, 버튼 disabled 상태, visibility 상태를 함께 수집한다.
- 화면이 "반응 없음"처럼 보이는 경우에는 서버 응답과 프론트 상태를 같이 확인한다.

## 발생 이력

- 2026-07-10: 승인 후 새 제안이 보이며 버튼이 막힌 것처럼 보이는 증상 확인.
- 2026-07-10: `visibilitychange` 미갱신 문제가 별도로 확인됨.
- 2026-07-10: 커밋 `4e8c39c` 계열로 상태 렌더링과 탭 재갱신 수정.
