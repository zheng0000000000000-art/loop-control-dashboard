# 스킬: 실행 검증

버전: 1 | 도메인: common | 트리거: 항상 | 대상: 구현한 기능이 실제로 동작하는지 확인하고 `docs/verification/*.md`를 쓸 때

## 절차

1. **코드 리뷰가 아니라 실제 실행으로 확인한다.** 서버를 띄우고(`dotnet run --project server` 또는 `-- measure <projectId>` CLI), 실제 HTTP 요청·CLI 호출로 결과를 본다. "코드상 이렇게 동작할 것이다"는 검증이 아니다.
2. **정상 상태에서 기준선을 먼저 확보한다.** 위반 0건, `suspendedTracks` 빈 배열, `overallStatus: completed` 같은 "아무 문제 없음" 상태를 측정해 시작점으로 남긴다.
3. **위반·장애를 의도적으로 주입한다.** 데이터 파일에 임시 값을 추가하거나(예: `ko.json`에 `__verificationTemp` 키), definition의 설정을 일시적으로 극단값으로 바꾸거나(예: `maxLoopIterations: 0`으로 가드레일 즉시 발동, endpoint를 존재하지 않는 포트로 바꿔 연결 실패 재현), 소스 파일의 주석 한 줄을 지운다. **주입 대상이 실제로 그 체크에 걸리는지 먼저 정규식·로직을 읽고 확인한다** — 접근제어자 없는 로컬 함수에 주석을 지웠다가 아무 일도 안 일어난 적이 있다(원인: 정규식이 `public/private/...`로 시작하는 줄만 본다).
4. **감지 결과를 실측한다.** run-log 이벤트, 생성된 proposal의 필드, ntfy 발송 내용(실제 topic으로 poll), 종료 코드, HTTP 상태 코드를 직접 확인하고 검증 문서에 그대로 옮긴다.
5. **원복하고 재확인한다.** 주입한 위반을 제거한 뒤 다시 측정해 "0건/aligned" 상태로 돌아오는지 확인한다. 원복하지 않고 세션을 끝내지 않는다.
6. **안 되는 것은 안 된다고 쓴다.** 도구가 타임아웃되면(`preview_screenshot`이 자주 그렇다) 대체 수단(`preview_inspect`로 좌표·계산된 스타일 확인)을 쓰고 그 사실을 문서에 남긴다. 검증 범위 밖(예: 사람의 물리적 기기 설정)은 "이 검증은 다루지 않는다"고 명시한다.
7. **결재는 사람 몫 — 검증 중 proposal이 생겨도 승인·거절하지 않는다.** 원복 측정으로 자연히 `superseded`가 되는 것을 확인하는 선에서 멈춘다.

## 지켜야 할 것

- ntfy 알림처럼 외부 서비스로 나가는 검증은 실제 서비스로 한다(로컬 스텁을 만들 수도 있지만, 최종적으로는 실제 엔드포인트로 최소 1회 실측한다).
- durationMs, HTTP 상태 코드, 이벤트 순서처럼 숫자·순서로 확인 가능한 것은 표로 옮긴다 — "잘 됐다"는 서술보다 표가 재현 가능하다.
- 검증 중 예상 밖의 결과(예: 알림이 두 번 온다, 자신이 방금 추가한 문구가 규칙을 위반한다)가 나오면 추측으로 넘기지 말고 run-log·측정 evidence를 추적해 실제 원인을 찾는다. 원인을 못 찾으면 "원인 불명"이라고 정직하게 쓴다.
- 임시로 바꾼 설정(endpoint, threshold, 토큰)은 검증이 끝나면 커밋 전에 원래 값으로 되돌렸는지 `git diff`로 반드시 확인한다.

## 완료 판정

- 검증 문서에 성공 경로와 실패/폴백 경로가 모두 실측 기록으로 들어 있다.
- 모든 임시 변경이 원복되어 `git status`가 의도한 파일만 보여준다.
- 결재 대기 상태로 남은 proposal이 있다면 그 사실과 이유가 문서에 명시돼 있다(숨기지 않는다).
