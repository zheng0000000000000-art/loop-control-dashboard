### 지표는 통과하지만 목적을 배신하는 경로
- `production-preconditions-not-default-true` self-test는 `tmpRoot`에 dotnet 프로젝트가 없어서 build 단계에서 실패시키는 것만 본다. launcher check와 high-risk check가 실제 검증인지, 기본 true가 아닌지만은 잡지 못한다. 지금 구현에 실재한다: `server/TrustOriginCli.cs:566-579`.
- high-risk 검증은 exit code만 본다. malformed envelope로 exit 2가 나와도 “검증 실패”로 닫히긴 하지만, 계약의 목적이었던 `trusted-human-receipt-required` 확인은 수행되지 않는다. 지금 구현에 실재한다.
- `RECOVERY` 또는 `REPLAY`가 나중에 실수로 허용되어도 `CheckHighRiskClosed`는 `PHASE_CHANGE`만 검사하므로 production precondition 10 검증은 통과 가능하다. 현재 `StateApplierCli` 본체는 세 kind를 모두 high-risk로 두고 있어 즉시 악용되지는 않지만, 검증 구현은 계약보다 좁다.
- malformed `.claude/settings*.json`은 “자동 launcher 비활성 확인 불가”인데도 skip된다. high-risk 버그가 고쳐진 뒤에는 이 경로로 launcher 확인 없이 record 생성까지 갈 수 있다.

### 테스트가 못 잡는 것
- high-risk 세 kind 각각에 대해 full envelope를 넣어야 한다. 예:
```json
{"schemaVersion":1,"transitionKind":"RECOVERY","transitionId":"hr-recovery","expectedPreStateSha256":"x","requestPath":"x","requestSha256":"x","effectiveAt":"2026-07-14T00:00:00Z","expectedPostStateSha256":"x","candidatePath":"x"}
```
기대: exit 1, reason/code `trusted-human-receipt-required`. 같은 입력을 `PHASE_CHANGE`, `REPLAY`로도 반복해야 한다.

- malformed launcher settings:
```json
{ "hooks":
```
기대: `trust-origin declare`가 선행조건 9 확인 불가로 거부. 현재 함수 단위로는 skip 후 true 가능.

- self-test의 `extra-known-exception`은 exit와 record 미생성만 본다. 지시서 완료 기준의 `extraKnownSubjects` 출력은 코드에는 있지만, 테스트가 제거 회귀를 잡지 못한다.

### 확인 못 한 것
- read-only 샌드박스라 `dotnet build`, `dotnet run --project server -c Release -- trust-origin --self-test`, `measure dev-pack`는 실행하지 않았다. 이 명령들은 `bin/`, `obj/`, dashboard data 또는 temp 산출물을 쓸 수 있다.
- `OUT-OF-SCOPE.txt`의 파일들은 누가 언제 썼는지 근거가 없어서 실행자 범위 이탈로 판정하지 않았다.
- `.claude/scheduled_tasks.lock`의 의미와 현재 자동 launcher 활성 여부는 이 코드 검수만으로 확정하지 않았다.
