### 판정
FAIL — `--self-test`가 실패 **코드 집합**만 비교해 같은 코드의 추가 실패/다른 subject 실패를 숨길 수 있다.

### 지표는 통과하지만 목적을 배신하는 경로
- 어떻게 통과시키는가: 기대 FAIL 케이스에 같은 failure code를 내는 추가 state id를 넣는다. 예: `pending-mismatch`에 `PENDING-OTHER-ID`를 하나 더 추가하고 log는 비운다.
- 무엇이 배신되는가: self-test는 “기대한 그 이유 하나”가 아니라 “같은 code 종류가 하나 이상 있었음”만 확인한다. fixture 오염/추가 실패를 녹색으로 숨긴다.
- 지금 구현에 실재하는가: 실재한다. `actualCodes`가 `HashSet<string>`이고 `SetEquals`만 한다 (`HandoffIntegrityCli.cs:398-403`).

- 어떻게 통과시키는가: failure subject를 잘못 내도 code만 같게 유지한다.
- 무엇이 배신되는가: `pending-mismatch`가 실제 state id `PENDING-REAL-ID`를 잡았는지, 엉뚱한 id를 잡았는지 self-test가 구분하지 못한다.
- 지금 구현에 실재하는가: subject를 비교하지 않으므로 가능하다 (`HandoffIntegrityCli.cs:412-413` 출력도 code만 기대값으로 둠).

확인한 긍정 사항:
- HarnessError는 더 이상 PASS로 뭉개지지 않는다. `expectHarnessErrors=false`와 실제 harness error가 다르면 `unexpected-harness-error`가 된다 (`HandoffIntegrityCli.cs:396-420`).
- `pending-nonok-zero` fixture는 추가됐고, `result!="ok" && exitCode==0` 입력이다 (`pending-nonok-zero/applier-log.jsonl:1`).
- `--pending-transition` 금지는 유지된다 (`HandoffIntegrityCli.cs:27-30`).
- `--self-test`는 경로/id 인자를 열지 않고 고정 fixture 경로를 쓴다 (`HandoffIntegrityCli.cs:363-365`).

### 확인 못 한 것
- 실제 `dotnet build`, `handoff-integrity --self-test`, `measure dev-pack`는 재실행하지 못했다. 이 검수 환경은 read-only이고, 현재 디렉터리는 `.git` 없는 리뷰 패킷이라 빌드/해시/기준 커밋 대조를 안전하게 재현할 수 없었다.
- `OUT-OF-SCOPE.txt`의 항목은 실행자 범위 이탈로 단정하지 않았다. 누가 언제 썼는지 근거가 없고, 사용자 지시상 확인 불가로 둔다.
