## ★★ 0순위 — 지금 게이트가 잠겨 있다 (23:0x 발생, 실측)

**`handoff-integrity`가 v9 계약을 모른다.**

- `server/Harness/HandoffIntegrityCli.cs:232`가 **단수 `blocker`**를 읽는다: `var blocker = ReadString(workstate, "blocker");`
- 그런데 `WORKSTATE.json`은 **v9 canonical 계약대로 복수 `blockers[]` 배열**이다(STATE-01이 신설. 필드 목록으로 확인함).
- **오늘 처음으로 `status=blocked`가 되자** 이 죽은 코드가 발화해서 **`blocked status requires a blocker field`**로 **exit 1**을 냈다.
- **결과: `state-transition`이 post-apply(handoff-integrity)에서 실패해 exit 1을 내고, 저장소 게이트 전체가 잠겼다.**

**고쳐라(§1과 같은 파일이다):**

1. `CheckBlockerConsistency`가 **`blockers[]` 배열**을 읽게 한다. `blocked`인데 배열이 비었으면 failure. `completed`인데 배열이 비어 있지 않으면 failure(기존 stale 검사의 취지 유지).
2. **단수 `blocker`는 더 이상 읽지 마라.** 하위 호환을 만들지 마라 — **그 필드는 WORKSTATE에 존재한 적이 없다**(STATE-01 검수에서 "한 번도 발화한 적 없는 죽은 경로"로 이미 기록됨).
3. **반증 시험**: 사본에서 `blocked` + `blockers` 비움 → **exit 1** / `blocked` + `blockers` 1건 → **exit 0** / `completed` + `blockers` 1건 → **exit 1**.

## ★★ 0순위-B — 게이트가 "다른 바이너리"를 검사한다 (23:4x 실측, 가장 위험한 것)

**`DiCompletionCheckCli.RunDotnetCommand`(:142)가 하위 검사를 이렇게 부른다:**

```
dotnet run --no-build --project server -- <command>      ← -c Release 가 없다 = Debug 기본
```

- 우리 관례는 **`-c Release`로 빌드**한다(`_header.md`·모든 지시서).
- 그런데 **게이트는 Debug 바이너리를 실행한다.** **Release만 빌드하면 게이트는 낡은 Debug 코드를 검사한다.**
- **실측 사고**: GUARD-03 실행자가 `HandoffIntegrityCli.cs`를 고치고 Release로 빌드했는데, `di-completion-check`가 **옛 Debug 바이너리**를 써서 `handoff-integrity` exit 1을 냈다. Debug를 따로 재빌드하고서야 PASS가 나왔다.
- **이건 "게이트 PASS가 방금 고친 코드에 대한 PASS가 아닐 수 있다"는 뜻이다.** 오늘 발견한 게이트 거짓말 중 **가장 위험하다.**

**고쳐라:**

1. `RunDotnetCommand`가 **호출한 구성과 같은 구성**으로 서브프로세스를 부르게 한다(`-c Release` 전달, 또는 실행 중인 어셈블리 위치에서 구성을 판별해 전달).
2. **`--no-build`인데 산출물이 소스보다 오래됐으면 fail-closed로 죽어라**(exit 2 + 이유). **조용히 낡은 바이너리로 판정하지 마라** — 그게 이 저장소의 고질병이다.
3. **반증 시험**: 소스를 고치고 **한 구성만 빌드**한 뒤 `di-completion-check` → **낡은 바이너리로 PASS를 주지 않는다**(fail-closed 또는 올바른 구성 실행).

## ★ §1 멱등 대조에 반드시 반영할 것 (오탐 방지 — 실측된 정상 케이스)

`post-apply` 단계(projection·handoff-integrity)가 실패하면 **WORKSTATE에는 전이가 적용되지만 `applier-log`에는 `result:"ok"`가 아니라 실패 사유가 기록된다.**
실측: `{"transitionId":"DI0004-BLOCKED-CODEX","result":"handoff-integrity-failed exit=1","exitCode":1}` — **`appliedTransitions`엔 있고 로그엔 `ok`가 없다.**

**이건 정상 동작이지 상태 오염이 아니다.** 멱등 대조를 단순히 "ok 집합 == appliedTransitions 집합"으로 짜면 **이 케이스를 오탐한다.**
→ **대조 규칙**: `applier-log`의 **모든 항목**(ok + 실패)을 기준으로 삼고, **로그에 아예 없는 id가 `appliedTransitions`에 있으면** failure. **로그에는 있는데(ok든 실패든) `appliedTransitions`에 없으면** failure. **판정 근거를 코드 주석과 verification 문서에 적어라.**

## 왜 하는가 — 오늘 게이트가 네 번 거짓말했다 (전부 실측)

| 사고 | 게이트 반응 |
| --- | --- |
| `state-transition` **CLI 배선이 통째로 사라졌다**(조율자가 allowlist 밖이라 격리) | `build` 0 · `gate-clean` 0 · `di-completion-check POST-COMMIT` **5/5 PASS** |
| DI-00-01 실행자가 WORKSTATE를 손복구 → `appliedTransitions`에서 전이 1건 누락 → **멱등이 실제로 깨졌다**(같은 id 재적용이 exit 0으로 통과, 검수자가 실증) | `handoff-integrity` **exit 0** (changedFiles 해시만 본다) |
| `run-executor.ps1`이 BOM 없는 UTF-8이라 PS 5.1이 한글 리터럴을 깨뜨려 **`FILE-CLAIMS.paths`가 항상 0**이었다 | `claim-check`는 **untracked를 못 봐서** 16회 연속 오탐만 냈다 |

**공통점: 검사는 있는데, 없어진 것을 보는 검사가 없다.**

> **hs-gate 2항 사전 확인 (네가 다시 검증하라)**: 이 하네스들이 볼 데이터는 **전부 실재한다** —

> **없는 데이터 위에 하네스를 세우지 않는다.**

## 할 일 (넷 다 **기존 하네스 확장**이다 — 신규 하네스 예산을 쓰지 않는다)

계약서(§1~§4)를 그대로 따르되, 핵심만 다시 적는다:

1. **`handoff-integrity` 멱등 대조** — `WORKSTATE.appliedTransitions[].id` ↔ `WORKSTATE.applier-log.jsonl`(`result:"ok"`) 집합 대조. 한쪽에만 있으면 **failure → exit 1**.
2. **`di-completion-check` CLI 계약 대조** — `docs/handoff/CLI-CONTRACT.json` 신설(**현재 실재 배선에서 열거해 생성**. 손으로 적지 마라). 계약에 있는데 배선에 없으면 **exit 1**, 배선에 있는데 계약에 없으면 **warning**.
3. **`GATE-MANIFEST.json`에 `scope-check`·`claim-check` 등재** — 등재되지 않은 하네스는 **아무도 돌리지 않는다.** 단 **판정할 수 없는 검사는 넣지 마라**(dirty 트리에서 늘 1이 나오는 성질을 감안하고, 넣지 않기로 했으면 **이유를 적어라**).
4. **`claim-check`에 `--untracked`** — `git grep -l {sym} -- server`는 untracked를 검색하지 않는다. 고치고 **오탐이 사라지는지 실측하라.**
