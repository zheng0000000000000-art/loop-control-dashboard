## 고칠 것 둘 + 하나

### 1. `--self-test`가 **실패의 원인**을 단언하게 하라

case 튜플을 확장한다:

```csharp
(name, pendingId,
 expectExemption      : bool,
 expectFailureCodes   : string[],   // 기대 Failure 코드 집합 (순서 무관, 정확히 이 집합)
 expectHarnessErrors  : bool)       // HarnessError가 나야 하는가 (기본 false)
```

판정:
- `HarnessErrors.Count > 0` 인데 `expectHarnessErrors == false` → **mismatch** (`unexpected-harness-error`).
  **입력이 깨진 것을 "검사가 잡았다"로 세지 마라.**
- `Failures`의 **코드 집합**이 `expectFailureCodes`와 다르면 → **mismatch** (기대/실제 둘 다 출력).
- `PendingExemptionApplied` 불일치 → mismatch (기존과 동일).

case별 기대 코드는 계약 §1-1 표를 그대로 따르라.

### 2. fixture 추가 — `pending-nonok-zero`

`result != "ok"` 인데 `exitCode == 0` 인 로그 항목:

```jsonl
{"transitionId":"PENDING-NONOK","result":"error-but-zero","exitCode":0,"at":"2026-01-01T00:00:01Z"}
```

`successfulLogEntries`는 **`result=="ok" && exitCode==0`**이므로 이건 성공이 아니다.
그러나 `allLogIdSet`에는 있다 → **면제 미적용** → `state-transition-not-logged`. **기대 exit 1.**
**현 코드는 올바르게 처리한다. 반증 자료가 없을 뿐이다.**

### 3. ★★ 자기 자신을 반증하라 — **완료 기준 5번이 이 작업의 핵심이다**

`pending-duplicate/workstate.json`을 **malformed JSON으로 일시 오염**시키고 `--self-test`를 돌려라.
**지금은 exit 0(PASS)이 나온다. 고친 뒤엔 exit 1(`unexpected-harness-error`)이 나와야 한다.**

**"고쳤다"의 증거는 "고치기 전엔 뚫렸다"를 보인 다음에만 성립한다. 건너뛰면 반려다.**

> **⚠ 시험 파일은 `$TEMP`에 사본을 만들어 거기서 오염시켜라.**
> 저장소 fixture를 직접 오염시킨 뒤 원복하는 것은 위험하다 —
> **검수자가 반증 시험 중 `docs/wiki` 42파일을 지운 사고가 있었다.** (git checkout으로 복구했다.)
> 부득이 저장소에서 했다면 **원복을 blob 해시로 확인하라**(`git hash-object` vs `git rev-parse HEAD:<path>`).
> **`git status`는 프록시다** — mtime만 바뀌어도 `M`이 뜨고, stat 캐시 때문에 놓칠 수도 있다.

## 목적 기준 (ADR-005)

**"계측기는 자기가 틀렸을 때 그것을 말할 수 있어야 한다."**
`--self-test`가 초록을 주는 이유가 **"기대한 그 이유로 실패했기 때문"**이어야 한다.
**"어쨌든 실패했으니 통과"는 목적 미달이다.**

우회로: `expectHarnessErrors`를 전 case에 true로 두기 · 기대 코드 집합을 부분집합 비교로 느슨하게 하기.
**둘 다 반려다. 자진 신고 없이 하면 반려다. 신고하면 감점이 아니다.**

**요구가 서로 모순되거나 빠진 게 있으면 완화하지 말고 보고해라.**
**직전 네 작업 중 세 번은 원인의 절반이 검수자가 준 요구의 모순 또는 공백이었다.** 이번에도 있으면 말해라.
