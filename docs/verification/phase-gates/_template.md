# HS-GATE 판정 템플릿 (v9 §0.4 정본)

> **이 파일은 형식·규칙 정의다.** `HS-GATE-P00.md`를 여기서 만들지 마라 — `DI-00-07`이 정확히 한 번 만든다.
> `HS-REVIEW-PXX-RN.md`(Phase 내 중간 검토)와 `HS-GATE-PXX.md`(Phase 최종 경계 판정) 둘 다 이 형식을 따른다.
> Harness와 Skill은 독립 판정한다 — 하나가 부적합해도 다른 하나는 제작할 수 있다.

## 판정 5종 (v9 §0.4)

| 판정 | 조건 | 추가 기록 필수 |
| --- | --- | --- |
| **즉시 제작 필수** | 총점 11~12 + gate-critical | 다음 Phase 전 검증 완료. gate-critical 사유 명시 |
| **기한부 제작** | 총점 11~12 + non-critical, 또는 총점 8~10 | 목표 Phase와 담당 WP 기록 |
| **기존 항목 확장** | 총점 8~10, 신규보다 기존 확장 우선 | 어느 Harness/Skill을 확장하는지 명시 |
| **보류** | 점수와 무관한 차단 사유 | **해소 조건·재판정 Phase 필수. 이유 없는 보류 금지** |
| **부적합** | 총점 0~4 | 기각 이유와 근거 데이터(FAIL ID) 기록 |

---

## 총점 규칙 (v9 §0.4)

```text
총점  0~4   → 부적합
총점  5~7   → 보류 또는 기존 항목 확장 검토
총점  8~10  → 기존 항목 확장 또는 기한부 제작
총점 11~12 + gate-critical → 즉시 제작 필수
총점 11~12 + non-critical → 다음 두 Phase 내 기한부 제작
```

**점수만으로 최종 결정하지 않는다.** 보안 위험·원본 변경 가능성·정책 미확정은 점수와 무관한 차단 사유다 → `보류` 또는 `부적합`.

## 예산 (v9 §0.4)

- **Phase당 신규 Harness 최대 2개 · 신규 Skill 최대 2개** (제작 의무 아님, 상한)
- **신규 제작보다 기존 Harness·Skill 확장을 우선 검토한다.**
- 예산 예외 가능: 출시 차단 불변식·보안 경계·Commit 안전성·복구 불가능 상태 방지

## Harness 판정 기준 (각 0=해당없음 / 1=부분충족 / 2=충족)

| 기준 | 0 | 1 | 2 |
| --- | --- | --- | --- |
| 반복 검증 가치 | 1회성 | 2회 반복 | 3회+ 또는 매 회차 |
| 결정 가능성 | 사람 해석 필요 | 부분 자동 | 완전 기계 판정 (PASS/FAIL 이분) |
| 장애 주입 가치 | 재현 불가 | 어려움 | 인위적 재현 자명 |
| 격리 가능성 | 부작용 큼 | 일부 상태 변경 | 읽기 전용·부작용 0 |
| 관찰 가능성 | 출력 불명확 | 결과만 | 왜 실패했는지 분해 출력 |
| 유지 비용 | 일회성 | 가끔 유용 | 게이트·루프가 직접 의존 |

Harness 제작 우선 조건:
- 동일 검증을 2회 이상 반복할 가능성
- 게이트 또는 출시 차단 조건을 자동 판정
- 장애 주입·동시성 재현이 수동보다 신뢰도 높음
- 회귀 시 중복 부작용·상태 불일치·권한 위반을 조기 탐지

## Skill 판정 기준 (각 0=해당없음 / 1=부분충족 / 2=충족)

| 기준 | 0 | 1 | 2 |
| --- | --- | --- | --- |
| 반복성 | 1회성 | 2~3 DI/WP | 여러 WP·Phase에서 반복 |
| 절차 안정성 | 절차 미확립 | 부분 고정 | 순서·확인 항목 완전 고정 |
| 입출력 명확성 | 불명확 | 부분 명시 | 입력·출력·형식 완전 명시 |
| 안전 경계 | 미정 | 부분 정의 | 금지사항·권한·중단 조건 완전 정의 |
| 판단 재사용성 | 개인 경험 | 부분 일반화 | 정책·체크리스트로 완전 일반화 |
| 도구화 가능성 | 불가 | 부분 자동화 | 스크립트·CLI·템플릿으로 완전 지원 |

Skill 후보 우선 검토 유형:
- Transition 복구와 reconcile
- 실패 사례 등록과 색인 갱신
- Artifact sealing·hash 검증
- Sandbox 생성·정리
- stale Approval 판정
- Shadow Artifact 승격
- Commit 전 안전성 검사와 rollback
- Legacy migration dry-run

---

## Skill 유형 (선언 필수 — v9 §0.4)

```text
procedural | assisted | executable
```

- `procedural`: 사람·LLM이 단계적으로 수행. 완전 자동화 불가 또는 불필요.
- `assisted`: LLM이 주도하되 사람 승인 포인트 포함.
- `executable`: 스크립트·CLI로 완전 자동화 가능.
