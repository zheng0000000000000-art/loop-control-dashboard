# 규칙의 근거 — 왜 그 규칙이 생겼는가

> **이 파일은 `CLAUDE.md`에서 분리됐다(2026-07-12, DIET-01).**
>
> `CLAUDE.md`는 **모든 에이전트의 모든 세션에 자동으로 실리고, 매 턴 다시 실린다.** 실측: 실행자 1회에 `CLAUDE.md`가 약 3,500토큰 × 턴 수만큼 청구된다.
> 그래서 **규칙(짧게)은 `CLAUDE.md`에, 근거(길게)는 여기에** 둔다. **규칙을 어긴 사람만 여기를 읽으면 된다.**
>
> **규칙을 지우지 않았다. 서사만 옮겼다.**

---

## 기준 파일(blueprint·workflow-definition) 변경이 왜 그렇게 엄한가

기준 파일은 `dashboard/data/*/blueprint.json`(목표치)과 `dashboard/data/*/workflow-definition.json`(가드레일·정책) **둘뿐**이다. 이 둘은 **커밋 대상**이고, 나머지 `dashboard/data`(measurement·run-log·patch-proposal·review-report·workflow-state)는 런타임 산출물이라 커밋하지 않는다.

**2026-07-11까지 `dashboard/data`를 통째로 제외하는 규칙 때문에 기준 변경이 git 이력에 하나도 남지 않았다.** 결재는 사람만 할 수 있는데, **그 결재의 결과물이 버전관리 밖에 있었다.** 측정 기준을 몰래 낮춰 게이트를 통과하는 것은 이 저장소의 금지사항 1호인데, 그걸 사후에 확인할 방법조차 없었던 것이다.

→ 기준 파일을 바꿨으면 `docs/handoff/BASELINE-CHANGES.md`에 **①주체(누가 승인했는가) ②근거 ③되돌리는 법**을 남긴다. 근거 없는 변경은 커밋되지 않고 HUMAN-INBOX에 '무단 변경 의심'으로 올라간다.

**아직 코드가 이걸 강제하지 않는다 — 프롬프트와 조율자 규칙뿐이다.** 하네스 후보다.

## 기록 파일을 왜 append만 하는가 (2026-07-12 실제 사고)

**대상**: `outputs/review-log.md`(조율자) · `outputs/reviewer-log.md`(검수자) · `docs/handoff/sessions/*`(코덱스) · `docs/handoff/HUMAN-INBOX.md` · `docs/handoff/BASELINE-CHANGES.md`.

조율자가 2026-07-11 회차에 `review-log.md`를 **통째로 읽어 통째로 다시 쓰다가**, 원래 깨져 있던 글자들을 **다시 인코딩해 파일을 오염**시켰다. 스스로 발견해 `git checkout`으로 이전 blob을 정확히 되돌리고 append로 다시 붙여 커밋을 교체했다(`fcb085b` 폐기 → `7dab8f2`). **이번엔 잡았지만 다음엔 못 잡는다.**

**근본 원인은 부주의가 아니다.** 이 저장소에는 **이미 깨진 한글이 박혀 있다**(예: `docs/handoff/WORKSTATE.json`의 note — `"IsWithinRoot(separator-bounded) ?ы띁"`). 그 바이트를 통째로 다시 쓰면 **조용히 바뀐다.** 게다가 깨진 글자는 U+FFFD가 아니라 **이중 디코딩된 정상 유니코드**라서 **자동 검출도 안 된다.**

→ 자기 기록을 고쳐야 하면 앞 내용을 수정하지 말고 **새 항목으로 정정을 덧붙인다**(기록은 이력이지 현재 상태가 아니다). 부득이 재작성해야 하면 **작업 전 해시를 남기고, 작업 후 `git diff`가 의도한 줄만 바뀌었는지 확인한다**(삭제 0줄).

## 검수자 기록을 왜 git에 넣는가 (ADR-003 보강)

`outputs/reviewer-log.md`는 **2026-07-12까지 untracked였다.** 조율자 로그는 커밋되는데 **검수자 로그만 버전관리 밖**이었다. 판정 근거가 git 밖에 있는 것은 위의 "기준 변경이 이력에 안 남았다"와 **같은 병**이다.

→ 검수자가 **직접 커밋**한다(관례 예외 ①에 준한다 — 자기 소유 기록 파일). 다른 주체는 이 파일에 쓰지 않는다(ADR-003 단일 기록자).

## 작업 보고에 왜 ①주체 ②하네스 ③스킬을 반드시 적는가

- **주체(actor)**: 같은 오류가 반복될 때 어느 주체 탓인지 추적하기 위함. **기록이 없으면 프록시로 추측하게 되고, 그러면 틀린다**(FAIL-2026-012 — 검수자가 커밋 접두사를 주체 서명으로 오인해 위반 22건을 날조할 뻔했다).
- **사용한 하네스**: 조율자가 이 목록을 보고 **직접 재실행해 대조한다.** 기록하지 않으면 검사할 수 없다. **자기보고는 증거가 아니다.**
- **참조한 스킬**: 어떤 지식을 썼는지.

## 지시 게이트 — 왜 추측으로 착수하면 안 되는가

FAIL-2026-013: 프롬프트가 인자 경계에서 잘려 **실행자가 지시서를 받은 적도 없이** 저장소를 읽고 **안전 보류 항목(FEAT-01)까지 구현**했다. 지시가 부족한데 추측으로 채우면 이런 일이 벌어진다.

→ 착수 전 검사: ①완료 기준이 검증 가능한가 ②대상 파일·범위가 특정되는가 ③기존 원칙·blueprint와 충돌하지 않는가. 부족하면 **선택지 딸린 질문**으로 되묻고 대기한다. 질문 없이 진행한 부분은 보고에 '추측 진행'으로 명시한다.

(FAIL-2026-013은 2026-07-12에 **구조적으로 제거**됐다 — 발사 래퍼가 프롬프트를 **파일로** 전달한다. 규약은 잊히지만 파일은 안 잊힌다.)

## 재발명 금지 — 계획서의 이름과 우리 이름이 다르다

`docs/plan/ALIGNMENT-v9.md`에 동등물 선언이 있다: `server/Harness/` = 계획서의 `harnesses/`, `skills/common/` = `skills/`, allowlist+`scope-check` = `FILE-CLAIMS`.

**이름이 다르다고 새로 만들지 마라.** 우리는 계획서를 읽지 않고 즉흥으로 만들다가 **같은 것을 다른 이름으로 재발명**했다.

## 왜 이 파일이 따로 있는가 (DIET-01의 근거)

실측(2026-07-12):

| 항목 | 토큰 |
| --- | ---: |
| `claude -p`로 "ACK만 출력해라" | **24,260** |
| 같은 방식으로 `measure` 한 번 실행 | **49,281** |
| 그중 실제 작업(입력 4 + 출력 189) | **193** |

**비용의 99.6%가 컨텍스트다.** 그리고 **캐시 읽기는 매 턴 앞부분 전체를 다시 읽는다** — 즉 **비용 ≈ 프리픽스 크기 × 턴 수**다.

`CLAUDE.md`는 모든 에이전트의 모든 세션에 자동으로 실린다. **거기서 1,000토큰을 덜면 20턴짜리 실행자는 20,000토큰을 아낀다.**

이 프로젝트의 첫 문장이 그것이다 — **"LLM은 적게 기억하고 적게 생성한다. 프로그램이 많이 기억하고 조립하고 검증한다."**
