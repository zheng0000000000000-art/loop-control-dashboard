# KNOWN-ISSUES — 오늘(2026-07-11) 세션에서 드러난 문제 정리

> 용도: 나중에 코덱스가 FAIL 위키로 자산화하거나, C# 오케스트레이터 설계의 요구사항으로 쓸 문제 목록. 각 항목은 증상·근본원인·재발방지 방향.

## I-1. 지시서↔실행 어긋남 (심각, 2회 반복 — 최우선)
- 증상: 헤드리스 sonnet을 특정 지시서로 발사했는데 다른 작업을 수행. ①FIX-01 발사(rec10)→FAIL-008(dispatch-templates) 수행. ②FIX-01 재발사(rec-fix01c/20900)→FIX-02(measure scope) 수행.
- 근본원인(가설): 발사 프롬프트로 준 지시서와 실행자가 저장소에서 스스로 고른 작업이 결속되지 않음. sonnet이 SONNET-QUEUE나 다른 컨텍스트를 보고 자기 판단으로 "다음 할 일"을 고르는 듯. `--dangerously-skip-permissions` + 자율 판단.
- 영향: 내가 의도한 작업(FIX-01 경로검증)이 계속 안 됨. 결과가 우연히 유용해도 통제 실패.
- 재발방지 방향: 발사↔실행을 task ID로 결속(발사 시 고유 ID, 완료 산출물이 그 ID를 명시). C# 오케스트레이터는 자식 프로세스에 단일 지시서만 주고 산출물 검증. 프롬프트에 "이 지시서 외 다른 작업 절대 금지"를 더 강하게. `원인 상태: 확정적 근본원인 미확정 — 재현·조사 필요`.

## I-2. FIX-01(경로검증 FAIL-006/007) 미해결
- 증상: sibling-prefix escape(StartsWith(root)) 취약점 수정이 여러 발사에도 완료 안 됨(FAIL 판정 반복).
- 원인: I-1(어긋남) + I-3(서버락 hang)에 막힘.
- 상태: **미해결. 보안 취약점 잔존.** Storage.cs·OutboxManager.cs origin에 separator-bounded 미적용.
- 다음: I-1·I-3 근본 정리 후 재작업. 서버 꺼진 상태에서 단일 지시서로.

## I-3. 헤드리스 실행자 vs 서버 exe 락 (hang 유발)
- 증상: sonnet이 escape 검증용으로 서버(dotnet run)를 띄움 → orphan으로 남아 bin exe 잠금 → 후속 build가 10회 재시도하다 hang(CPU idle).
- 부수: 그 orphan이 라이브 5173 포트까지 점유(정상 서버 밀어냄).
- 재발방지: 검증은 서버 안 띄우고 인프로세스/CLI로. 부득이하면 임시 포트+즉시 종료. 빌드는 `-o 분리경로` 또는 서버 정지 상태에서. (FIX-01 지시서에 반영했으나 I-1 때문에 미검증)

## I-4. escape-check orphan의 라이브 포트 점유 (해결됨 — 서버 재시작 필요)
- 서버 껐음(orphan 정리, 빌드 락 해소). 라이브 대시보드 재시작 필요(반입 대기 2건 결재 위해).

## I-5. last-reviewed 포인터 갱신 누락 (경미, 완화됨)
- 조율자가 커밋 안 한 주기엔 포인터 미갱신 → 여러 주기 stale. 오케스트레이터가 갱신하게 했으나 그 오케스트레이터를 안전상 비활성화. 현재 수동 갱신.

## I-6. 무인 자동화 안전 (설계 결정, 완화 완료)
- 무인 자동 push + permission-bypass 에이전트 무한 spawn은 위험 → 조율자를 "검수·로컬커밋·리포트만, push·발사는 사람 배치 게이트"로 재설계. 오케스트레이터·자동발사 비활성화. **결론: 되돌리기 어려운 행동(외부 push, 에이전트 발사)은 무인으로 열지 않는다.**

## I-7. (해결됨) StartTime 데드락
- claude.exe StartTime으로 sonnet 실행 판정 → Cowork 앱 등이 항상 떠 있어 영구 "실행중" → 발사 데드락. PID 파일 기반으로 수정.

## I-8. CRLF 데드락 — 큐가 다음 항목으로 넘어가지 못한 진짜 원인 (해결됨 2026-07-11)

- 증상: 조율자가 매 회차 "server/docs 변경 없어 커밋 없음" + "FEAT-02 발사 대기"를 동시에 보고하며 큐가 #3에서 영구 정지.
- 원인: 어떤 도구가 `server/*.cs` 5개(DevPackMeasures·OllamaExecutor·OutboxManager·Program·Storage)를 **LF→CRLF로 재작성**. 내용 변경은 0(CR 제거 시 HEAD와 SHA 완전 일치, 5개 전부 검증). 저장소에 `.gitattributes`가 없고 `core.autocrlf`도 unset이라 정규화 장치가 전무 → git이 이 5개를 영구 "수정됨"으로 인식.
- 데드락 구조: 발사조건①(`git status --porcelain -- server` 비어야 함)이 **영원히 거짓** → 발사 불가. 동시에 조율자는 내용 기준으로 "커밋할 것 없음"이라 정직하게 판단 → 커밋도 없음. **커밋할 것도 없고 dirty라서 발사도 못 하는 교착.**
- 기간: mtime 기준 OllamaExecutor.cs는 07-09부터 CRLF. 즉 발사조건①은 며칠째 거짓이었다(그동안 FIX-01/02가 돈 것은 사람이 직접 발사했기 때문).
- 조치: `.gitattributes` 신설(`* text=auto eol=lf`, `*.cs text eol=lf`) + `git add --renormalize .` → server/ clean 회복. 내용 손실 0.
- 교훈(ORCH 사양 입력): **게이트 판정을 줄바꿈 같은 표현 차이에 의존시키지 말 것.** 오케스트레이터의 "clean" 판정은 정규화된 내용 해시로 해야 하며, 도구가 파일을 다시 쓰는 것만으로 게이트가 영구 잠기면 안 된다. 또한 조율자가 "커밋 없음"과 "발사 대기"를 동시에 보고하면 **그 둘이 같은 원인일 수 있음**을 의심하도록 리포트에 교차검증을 넣어야 한다.
- 주의: server/가 clean해졌으므로, 조율자의 자동발사가 활성화되어 있었다면 다음 회차에 FEAT-02가 무인 발사될 수 있는 상태다. 현재 조율자는 "발사는 사람 승인 대기로만 기록"이므로 무인 발사는 일어나지 않는다 — 이 전제가 바뀌면 재검토할 것.

## I-9. 문서 부분 쓰기로 인한 잘림 (복구됨, 원인 미확정)

- 증상: `docs/STATUS.md`(4번 항목 중간에서 절단)와 `docs/handoff/SONNET-QUEUE.md`("QUOTA_"에서 절단) 두 문서가 파일 끝이 잘린 채 워킹트리에 존재.
- 원인 추정: 쓰기 도중 프로세스 중단(orphan 정리 시점의 kill 등)으로 인한 **비원자적(non-atomic) 부분 쓰기**. .cs 파일들은 잘리지 않았음(줄 수·마지막 줄 무결).
- 조치: HEAD 기준으로 두 문서 tail 복구(커밋 9e2268b). 이후 재발 없음 확인.
- 권고: 루프가 문서를 쓸 때 **임시파일 + 원자적 rename**으로 쓸 것. 핵심 상태파일(WORKSTATE.json)이 같은 식으로 잘리면 피해가 크다.
