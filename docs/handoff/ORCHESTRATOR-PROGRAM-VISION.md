# 오케스트레이터 프로그램화 — 루프를 C#으로 (장기 목표)

> 상태: **Deferred / 방향 기록.** 현재 루프가 요구사항을 학습 중. 하네스/스킬 자산화가 충분히 쌓이고 현재 루프가 안정된 뒤 착수한다.
> 로드맵: Axis1 Runtime — Scheduler / Local·Cloud Orchestration.
> 고정점(불변): 결재·기준 변경·이양 결정은 착수 후에도 여전히 사람. 오케스트레이터 프로그램은 그 게이트를 대행하지 않는다.

## 현재 (프로토타입)
조율자·코덱스·발사 루프가 **스케줄 태스크 + PowerShell**로 임시 구현돼 있다:
- 조율자(5분): 검수·커밋·큐 발사 — scheduled task 프롬프트 + PowerShell.
- 코덱스(15분): QA·E2E·HS-GATE — scheduled task.
- 발사: `claude -p` 헤드리스 + PID 파일로 실행 감지.
한계: 상위 모델(스케줄 실행 시 Claude) 토큰 소모, 취약(데드락·PID 파일·프로세스 감지·혼선·빈 커밋).

## 목표
이 오케스트레이션을 서버 C# 프로그램(`OrchestratorService`)으로 내재화한다. **실행자(sonnet/codex CLI)만 LLM**이고, 조율·검수·커밋·큐·발사·감지는 전부 프로그램이 한다.
- Scheduler: C# 백그라운드 타이머(5분/15분).
- 발사·감지: 서버가 executor 자식 프로세스를 직접 관리(PID 파일 불필요, 데드락 불가).
- 검수: 서버가 build/verify-behavior/measure/하네스를 호출·판정.
- 커밋: 기존 `GitDataCommitter` 확장.
- 큐: 서버가 SONNET-QUEUE를 관리(상태 정합 자동).
- 실행자 사다리·할당량: #13과 통합(코덱스↔claude-code 강등).

이점: 토큰 0(상위 모델이 조율 안 함) · 견고(프로세스 직접 관리) · 결정론 · 자기완결(외부 스케줄 의존 제거).

## 왜 나중인가 — 현재 루프가 명세를 만든다
지금 겪는 모든 결함이 C# 설계의 직접 입력이다:
- 데드락(claude.exe StartTime 오판) → 오케스트레이터는 자식 PID를 직접 소유하므로 이 문제 자체가 없음. **요구사항: 실행 감지는 소유한 프로세스 핸들로.**
- 지시서 어긋남(rec10이 다른 작업 수행) → **요구사항: 발사↔완료를 task ID로 결속, 산출물 검증.**
- 빈 커밋·큐 상태 정합 → **요구사항: 상태 기계로 큐 관리.**
- last-reviewed 포인터 갱신 누락 → **요구사항: 단일 진실 원본(WORKSTATE)에서 파생.**
즉 프로토타입을 돌리는 것 자체가 사양서 작성이다. FAIL 위키·review-log가 그 사양의 근거 데이터가 된다.

## 단계
1. (현재) 스케줄+PowerShell 루프 — 프로토타입·엣지케이스 학습.
2. 검수·QA 로직 CLI 하네스화(verify-behavior·measure·path-guard·call-integrity·e2e-usage 등) — 오케스트레이터가 호출할 부품. **지식 승격 파이프라인(KNOWLEDGE-PROMOTION)이 이 단계를 채운다.**
3. #13 실행자 사다리·할당량 원장 — 오케스트레이터의 발사 계층.
4. `OrchestratorService` C# 구현 — 타이머·큐·발사·검수·커밋을 코드로. 스케줄 태스크 은퇴, review-log는 서버 이벤트로.
5. 검증: 오케스트레이터가 무인으로 큐를 소진하고, 사람 게이트(결재·기준)만 남는지 E2E.

## 최종 그림
사람이 blueprint(기준)와 큐 우선순위·결재만 정하면, C# 오케스트레이터가 실행자(LLM)들을 발사·검수·커밋·승격하며 스스로 돈다. 지금 상위 모델(나)이 하는 조율 역할이 프로그램으로 내려가고, 상위 모델은 큐 채우기·판단 같은 진짜 판단만 남긴다 — "LLM은 판단, 프로그램은 조립·검증·기억"의 최종형.
