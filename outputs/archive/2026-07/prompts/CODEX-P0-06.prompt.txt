### 데이터 관문은 열렸다 (hs-gate 2항)

`docs/handoff/FILE-CLAIMS.json` 이 **실재한다.** 발사 래퍼가 실행자를 띄울 때 지시서의 `## 허용 파일 (allowlist)`을 파싱해 **claim으로 등록**하고, 종료 시 `released`로 바꾼다.

```json
{ "schemaVersion": 2,
  "claims": [
    { "claimId": "LEDGER-04-34288", "actor": "sonnet", "taskId": "LEDGER-04",
      "paths": ["server/OllamaExecutor.cs", "server/Program.cs", "..."],
      "allowlistSource": "docs/handoff/queue/directive-LEDGER04-metricid-normalization.md",
      "pid": 34288,
      "claimedAt": "2026-07-12T04:00:09+09:00",
      "expiresAt": "2026-07-12T06:00:09+09:00",
      "releasedAt": "2026-07-12T04:10:34+09:00",
      "status": "released", "exitCode": -1 },
    { "claimId": "SMOKE-01-11612", "actor": "sonnet", "taskId": "SMOKE-01",
      "paths": [], "allowlistSource": null,
      "pid": 11612, "status": "released", "exitCode": 0 } ] }
```

### ★ `allowlistSource`가 `null`이면 **"allowlist를 모른다"**는 뜻이다 — 빈 목록과 다르다

발사 래퍼의 첫 판에 버그가 있었다: 지시서를 못 찾으면 `paths`를 **`null`**로 적었다(PowerShell이 빈 배열을 `$null`로 접는다). 검수자가 잡아 고쳤다.

- **`paths: [], allowlistSource: <지시서 경로>`** → **"이 작업은 아무 파일도 잡지 않는다"**(진짜 빈 목록).
- **`paths: [], allowlistSource: null`** → **"allowlist를 모른다"**(지시서를 못 찾았다. 스모크 테스트 등).

**이 둘을 같게 취급하지 마라.** `allowlistSource: null`인 claim에 대해 "아무 파일도 안 잡았으니 충돌 없음"이라고 판정하면 **모르는 것을 안전으로 오판**하는 것이다. **경고로 출력해라.**

### 만들 것 (세 가지)

1. **충돌 검출** — `git status`의 변경 파일이 **다른 주체의 `status: active` claim과 겹치면 exit 1** + 겹친 파일·주체·taskId·pid를 출력한다.
   - 같은 주체(자기 claim)와 겹치는 것은 **정상이다.** 그건 자기 작업이다.
   - **되돌리지 마라. 삭제하지 마라. 프로세스를 죽이지 마라. 검출·보고만 한다.**
2. **stale claim 검출** — `pid`가 **죽었는데** `status: active`면 경고(`expired` 후보)로 출력한다. **자동으로 지우지 마라** — 사람이 판단한다.
   - 판정 근거는 **PID 생존 여부**다. `expiresAt` 시각은 보조 근거일 뿐이다. **시각으로만 판단하면 오래 도는 실행자를 죽은 것으로 오인한다.**
3. **기존 기능 유지** — 지시서 allowlist 대조(사후 검출)는 **그대로 둔다.** 확장이지 대체가 아니다.
