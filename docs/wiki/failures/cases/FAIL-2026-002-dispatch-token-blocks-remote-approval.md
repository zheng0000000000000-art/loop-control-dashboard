# FAIL-2026-002 — 토큰 미설정 시 원격 반입/승인이 401로 막힘

- 상태: 완화됨
- 최초 발생일: 2026-07-10
- 최근 발생일: 2026-07-10
- 관련 DI: 지시서 #16(모바일 승인 토큰 개선)
- 구성요소: remote-token, dispatch-auth, dashboard-approval
- failureClass: config_side_effect, design_learning
- 심각도: high

## 발생 상황

`RemoteActionToken`이 빈 문자열이면 dispatch·반입 계열이 로컬에서도 무조건 401을 반환했다. 이는 코드 실행 경로를 토큰 없이 열지 않는 안전장치였다. 이후 사용자가 반입을 위해 토큰을 설정하자 일반 proposal 승인 같은 모든 POST도 토큰을 요구하게 됐다.

## 관찰된 증상과 영향

프론트는 401 응답을 받으면 `window.prompt`로 토큰을 묻고 메모리에만 저장했다. 모바일 원격 환경에서는 토큰을 매번 다시 입력해야 했고, 상황에 따라 조용한 실패처럼 보였다. 그 결과 사람이 모바일에서 결재를 진행하지 못해 승인 병목이 생겼다.

## 발생 이유(직접·근본·기여, 미확정은 가설 표시)

- 직접 원인: 서버가 토큰 설정 시 모든 POST에 토큰을 요구했고, 프론트의 토큰 저장은 메모리 한정이었다.
- 근본 원인: dispatch·반입 보호를 위해 둔 토큰 정책의 부작용이 일반 모바일 결재 UX까지 확장됐다.
- 기여 요인: 모바일 브라우저에서 prompt 기반 재입력 흐름은 상태가 보이지 않고 실패 복구가 어렵다.
- 미확정: 모바일별 브라우저 저장 정책 차이가 실패 빈도에 얼마나 영향을 줬는지는 별도 계측하지 않았다.

## 검토한 해결 방법

- 서버에서 일반 proposal 승인 POST만 토큰 예외로 둔다.
- 서버 게이트를 유지하고 프론트에서 토큰 상태를 명확히 표시한다.
- 토큰을 매번 prompt로 입력하게 유지한다.

## 선택한 해결 방법

지시서 #16로 서버 게이트는 의도된 사양으로 유지하고, 프론트에서 토큰을 `localStorage`에 영속화했다. 또한 토큰 상태 UI와 401 오류 명확화를 추가해 사용자가 현재 인증 상태를 알 수 있게 했다.

## 판단 기준

- 코드 실행·반입 경로는 토큰 없이 열지 않는다.
- 모바일 결재자는 토큰을 반복 입력하지 않아야 한다.
- 실패 시 401 원인이 화면에서 명확해야 한다.

## 검증 결과

지시서 #16 산출물과 검증 문서가 생성됐다. 이 사례에서는 제공된 사실만 근거로 삼으며, 서버 게이트 무수정과 프론트 토큰 영속화·상태 UI·401 명확화가 완화책으로 기록됐다.

## 재발 방지

- 토큰이 필요한 환경에서는 프론트가 토큰 상태를 보여준다.
- 토큰은 메모리만이 아니라 `localStorage`에 저장한다.
- 401을 조용한 실패로 처리하지 않고 사용자가 이해할 수 있게 표시한다.

## 후속 작업과 잔여 위험

- 잔여 위험: 토큰이 평문으로 저장된다.
- 수용 근거: 로컬 개인 도구와 Tailscale 사설망 사용 전제에서 완화책으로 받아들였다.
- 후속 관찰: 공용 장치나 공유 브라우저 사용 시 별도 보호가 필요하다.

## 발생 이력

- 2026-07-10: `RemoteActionToken` 미설정 상태에서 dispatch·반입 계열 401 확인.
- 2026-07-10: 토큰 설정 후 일반 proposal 승인 POST도 토큰을 요구하는 부작용 확인.
- 2026-07-10: 지시서 #16로 프론트 토큰 영속화와 상태 UI 완화책 작성.
