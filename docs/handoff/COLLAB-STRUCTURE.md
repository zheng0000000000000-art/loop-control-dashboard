# 다중 실행자 협업 구조 — sonnet · 코덱스 · 오케스트레이터

> 목적: 세 실행자의 협업을 채팅 릴레이가 아니라 **저장소를 메시지 버스로 쓰는 비동기 구조**로 고정한다. 코덱스는 CLI가 없어 채팅 맥락에 기댈 수 없으므로, 모든 인수인계는 저장소 파일로 한다(v9 §0.5 준수).
> 근거 사건: 병렬 통제 실패(FAIL-2026-004). 이 구조가 그 재발을 막는 장치다.

## 1. 실행자와 역할

| 실행자 | 채널 | 역할 | 쓰기 영역 |
| --- | --- | --- | --- |
| **sonnet** | 헤드리스 CLI (자동 발사) | 구현·리팩토링·버그 수정 | `server/` (+지정 시 dashboard/) |
| **코덱스** | 사용자 릴레이 (수동) | QA·버그헌팅·재현·실패 자산화·1차 검수 | `docs/qa/`, `docs/wiki/failures/` |
| **오케스트레이터/검수자** (Claude) | 이 세션 | 작업 분해·지시서 작성·독립 검수·커밋·발사·조율 | `docs/directives/`, `docs/verification/`, `docs/handoff/`, 읽기 전체 |

**결재·반입·기준 변경·이양 결정 = 항상 사람.** 실행자는 그 게이트를 대행하지 않는다.

## 2. 영역 소유 (FILE-CLAIMS 간이판)

위 표의 쓰기 영역은 **배타적**이다. 두 실행자가 같은 파일·디렉터리를 동시에 쓰지 않는다. 지시서마다 "영역 격리" 절로 재확인한다. 위반 = FAIL-2026-004 재발 → 검수에서 FAIL.

동일 영역(예: server/) 작업은 **순차**로만(동시 발사 금지). 다른 영역(server/ ↔ docs/qa/)은 병렬 안전.

## 3. 핸드오프 흐름 (저장소 = 메시지 버스)

```
오케스트레이터: 작업 분해 → 지시서(docs/directives 또는 outputs) → 실행자 영역에 배정
        │
        ├─ sonnet: server/ 구현 → 커밋 → [검수] → 반영
        │
        └─ 코덱스: 버그 발견 → docs/wiki/failures FAIL 등록  ← 이게 구현 입력이 됨
                        │
                        └─→ 오케스트레이터가 FAIL을 읽고 → sonnet에 수정 지시 → 순환
```

- **코덱스→sonnet 간접 협업**: 코덱스가 등록한 FAIL 위키가 sonnet 수정 지시의 근거가 된다. 나(오케스트레이터)가 중개하되, 근거는 저장소 파일이라 추적 가능.
- **검수**: 모든 커밋은 `VERIFY-PROTOCOL-universal.md`로 검수. 코덱스도 1차 검수자가 될 수 있다(검수 병목 분산).

## 4. 코덱스 작업 큐 — `docs/handoff/CODEX-QUEUE.md`

오케스트레이터가 채우고, 코덱스가 위에서부터 픽업한다. 형식:

```
| 순번 | 작업 | 지시서 | 영역 | 상태(대기/진행/완료) |
```

현재 큐(초기):
1. S-01/S-02 경로 escape 재현 (directive-CODEX-3) — docs/qa+wiki — **진행**
2. 버그헌트 라운드2: 리팩토링(R-01~04) 후 CliRouter/InboxBuilder/CycleSummary/MeasurementService 호출부 정합성 헌트 — docs/qa — 대기
3. 검수 위임 시범: 다음 sonnet 커밋 1건을 VERIFY-PROTOCOL로 코덱스가 1차 검수 → docs/qa/review-*.md — 대기

## 5. 통신·세션 규칙

- 코덱스는 작업 시작 시 **3개 문서**를 읽는다: `CODEX-ROLE-bug-hunter.md`(역할), 해당 작업 지시서, `VERIFY-PROTOCOL-universal.md`(검수 시). 채팅 맥락에 의존하지 않는다.
- 작업 종료 시 `docs/handoff/sessions/SESSION-<날짜>-codex-NNN.md`에 남긴다: 한 것/미완/발견/다음 픽업 후보. 다음 실행자(또는 다음 코덱스 세션)가 이걸로 이어받는다.
- 커밋·push는 하지 않는다. 파일만 만들고 사람/오케스트레이터가 반영·커밋.

## 6. 이 구조가 지향하는 것 (북극성 연결)

궁극 목표는 사람이 기준만 정하고 AI들이 서로를 다루는 런타임이다. 이 협업 구조는 그 최소 실물이다: 구현(sonnet)·QA(코덱스)·검수/조율(오케스트레이터)이 **저장소를 통해** 비동기로 물리고, 사람은 결재 게이트와 큐 우선순위만 쥔다. 지금은 오케스트레이터(나)가 중개하지만, 큐·핸드오프·검수 프로토콜이 문서로 고정될수록 중개 비중이 줄어든다.
