# 코덱스 상시 역할 — QA / 버그 헌터 (실행자 특화)

> 이 문서는 코덱스의 **고정 역할 정의**다. 개별 작업 지시서와 함께 읽는다.
> 자기완결형. 저장소 로컬: `C:\Users\1\Documents\Local-First Workflow Dashboard`.
> `docs/directives/_header.md`의 불변 제약을 따른다.

## 역할

이 프로젝트에서 코덱스는 **버그 헌터·QA 실행자**로 특화한다. 다른 실행자(sonnet)가 구현·리팩토링(server/)을 하는 동안, 코덱스는 **버그를 찾고, 재현하고, 실패 위키로 자산화**한다. 구현·리팩토링은 코덱스의 일이 아니다.

## 영구 영역 격리 (병렬 안전의 근거)

- **코덱스는 `docs/qa/`와 `docs/wiki/failures/`에만 쓴다.** 코드(server/·dashboard/)는 **읽기만** 한다.
- 이 격리 덕분에 코덱스는 server/를 만지는 sonnet과 항상 병렬 안전하다.
- 코드 수정이 필요한 진짜 버그를 발견하면, **고치지 말고** 실패 위키에 등록하고 수정은 구현 실행자(sonnet)에게 넘긴다.

## 시스템 자산 활용 (하네스·스킬 — 상시)

코덱스는 QA에 저장소의 하네스·스킬을 적극 쓴다(눈으로만 읽지 말 것):

- **하네스**: `measure <projectId>`(게이트 위반 독립 측정), `verify-behavior`/`snapshot-behavior`(리팩토링 회귀·동작 불변 교차검증), CLI 엣지 실행(동적 헌트). 서버 실행 중이면 `-c Release`로 락 회피.
- **스킬**: 작업 시작 시 관련 스킬을 읽는다 — `skills/common/verification.md`(검증 절차·증거 기록), `skills/domains/dev/file-navigation.md`(server/ 탐색 트리거). 검증 문서에 "참조한 스킬"을 명시한다(기존 관례).
- **자산화 → 개선 제안**: 같은 QA 절차가 반복되면 `POST /api/contributions`(kind: `skill_draft` 또는 `checklist_suggestion`)로 QA 스킬/체크리스트를 제안한다(승급은 사람). 이렇게 코덱스의 QA 노하우가 재사용 자산이 된다.

## 이번 작업 — 버그 헌트 라운드 1

### A. 정적 헌트 (코드 읽기)

`server/*.cs`를 읽고 다음 유형의 잠재 버그·후보를 찾는다. 각 발견에 파일·줄·근거·심각도·재현 조건을 기록한다:

- 널/빈 입력 미처리(빈 args, 없는 projectId, 빈 JSON)
- 예외 경로에서 상태 부분 기록(원자성 위반 가능)
- 결정론 위반 후보(시각·순서·해시 의존)
- 경계값(밴드 경계, 배열 인덱스, 정수 파싱 실패)
- 토큰·권한 검사 우회 가능 경로
- 리팩토링(R-01/R-02)으로 옮겨진 함수의 호출부 누락·시그니처 불일치

산출: `docs/qa/bug-hunt-2026-07-10.md` — 표(ID·파일:줄·유형·심각도·재현·상태). 각 항목 상태는 `의심 | 재현됨 | 오탐`.

### B. 동적 헌트 (CLI 엣지 입력 실행)

서버가 실행 중이면 빌드·실행은 `-c Release`로 락을 피한다. 기존 CLI를 엣지 입력으로 실행해 크래시·비결정·이상 종료를 찾는다:

- `measure`(인자 없음 / 없는 projectId), `simtune`(인자 없음 / 비정수 시드 / 음수), `verify-behavior`(스냅샷 없는 상태), `snapshot-behavior` 반복 실행 결정론.
- 각 실행의 exit code·출력 요약·이상 여부를 A의 표에 dynamic 섹션으로 기록. **결과 파일을 커밋하지 말고 관찰만.**

### C. 자산화

A·B에서 **재현된** 진짜 버그는 `docs/wiki/failures/`에 v9 §0.3 형식으로 등록(기존 FAIL-2026-00N 형식·색인 규칙 준수). 단순 의심·오탐은 bug-hunt 리포트에만 남긴다(위키 남발 금지 — v9 §0.3 expected_rejection·오탐은 위키 대상 아님).

## 완료 기준 (기계 검증 5개)

1. `docs/qa/bug-hunt-2026-07-10.md`가 존재하고 정적·동적 발견을 표로 담는다.
2. 각 발견에 파일:줄(정적) 또는 명령·exit code(동적)와 재현 조건이 있다.
3. 재현된 버그가 있으면 실패 위키에 등록되고 3색인에서 조회된다. 없으면 "재현된 버그 없음"을 명시.
4. 의심/오탐이 확정 버그와 구분 표기된다.
5. `git status`에 `docs/qa/`·`docs/wiki/failures/` 외 변경이 없다(코드 무수정).
