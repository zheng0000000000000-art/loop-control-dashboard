## 왜 하는가 (실측)

**독립 재개 시험이 FAIL했다.** L0만 읽은 실행자가 이렇게 답했다:

```
PHASE=P0-04   ← 거짓. 실제로는 P0-06까지 끝났다
STATUS=verifying
NEXT=부족: 무엇을 검증 중인지, 어느 하네스를 돌려야 하는지 이 파일만으로는 알 수 없다.
```

**원인은 Projection의 표현력이 아니다. 상태를 전이시키는 주체가 없다는 것이다.**

- LEDGER-04 실행자가 WORKSTATE를 `verifying`으로 두고 나갔다.
- 그 뒤 **코덱스가 P0-05·P0-06을 끝냈는데 아무도 WORKSTATE를 갱신하지 않았다.** 코덱스는 WORKSTATE에 쓰지 않는다.
- **갱신 주체가 없다. 이건 담당자 실수가 아니라 시스템이 허용한 정상 경로다.**

## 할 일

### 1. WORKSTATE를 v9 canonical 계약으로 (새 이름 만들지 마라)

```json
{
  "schemaVersion": 3,
  "phaseId": "P00",
  "wpId": "WP-00",
  "diId": "DI-00-0Y",
  "status": "blocked",
  "blockers": [],
  "nextActions": []
}
```

- **v9 필드명을 그대로 쓴다**(`docs/plan/AI-RUNTIME-REFACTOR-MICRO-DIRECTIVES-v9.md:522-549`). **`WP-P0-SAFETY` 같은 새 이름 금지.**
- 기존 `blocker`(단수)는 **없으니 마이그레이션할 것도 없다.** `blockers` 배열을 **신설**한다.
- **⚠️ `diId`를 `DI-00-07`로 올리지 마라.** 그건 **"DI-00-01~06이 전부 끝났다"는 경계 주장**이다. **적합성 행렬(검수자 작성 중)이 증명하기 전까지, 가장 이른 미충족 DI가 현재 `diId`다.**
  → **이번 작업에서는 `diId`를 바꾸지 마라.** 스키마와 전이 경로만 만든다. **값 결정은 검수자·사람 몫이다.**
- **비canonical 값(`P0-07`·`LEDGER-04` 등)을 canonical 필드에 넣지 마라.** 로컬 큐 별칭은 `notes`에 둔다.

### 2. `ProjectionCli`가 `blockers[]`를 읽고 **복수로** 출력하게

- `RUNTIME-INDEX.md`(L0)와 `HANDOFF.md` 둘 다.
- **L0는 20줄 이내로 유지한다**(v9 §0.6). L0는 **모든 재개 세션이 읽는 프리픽스**다 — 여기가 커지면 **모든 세션의 비용이 커진다.**
- L0에 반드시 담아라: **Phase · WP · DI · status · blockers · nextActions**. 그래야 새 세션이 **지어내지 않고** 답할 수 있다.

### 3. ★ `state-transition` — **WORKSTATE의 유일한 writer**

`server/StateApplierCli.cs` 신규 + `server/Cli/CliRouter.cs` 배선.

**입력**(전부 파일/인자로 받는다. **자연어 보고서를 해석해 완료를 추론하지 마라**):

```
--transition-id <고유 ID>
--expected-workstate-sha256 <현재 WORKSTATE의 해시>
--request <transition request JSON 경로>
--verdict <독립 검증 결과 JSON 경로>     (completed 전이에 필수)
--human-decision <사람 결정 JSON 경로>   (Phase gate·기준 변경에만)
```

**규칙 (하나라도 어기면 쓰지 않는다):**

1. `expected-workstate-sha256`이 **현재 WORKSTATE와 다르면 쓰지 않는다**(낙관적 동시성. 남이 먼저 바꿨다는 뜻이다).
2. **같은 `transition-id`는 한 번만 적용한다**(멱등).
3. canonical ID 패턴과 status enum을 **검증한다.**
4. **`blocked`인데 `blockers`가 비어 있으면 실패.**
5. **`waiting`·`in_progress`·`verifying`·`blocked`인데 `nextActions`가 비어 있으면 실패.** ← **독립 재개가 FAIL한 바로 그 지점이다.**
6. **`completed` 전이에는 독립 verification PASS가 필요하다.** **생산자 자기보고만으로 완료가 될 수 없다.**
7. **Phase gate 전이에는 사람 결정이 필요하다.**
8. **어떤 검증 실패에서도 기존 WORKSTATE를 손상시키지 마라.**

**쓰기 절차 (순서 고정):**

```
현재 해시 대조 → 입력 검증 → candidate를 임시 파일에 생성 → candidate 재검증
  → atomic replace → projection 실행 → handoff-integrity 실행
  → 후속 단계가 실패하면 transaction failure를 명시적으로 기록하고 중단한다
```

**중간 실패를 green으로 바꾸지 마라.** 적용 상태(`transitionId` + 결과)를 남겨 **다음 실행이 불완전 적용을 탐지**할 수 있게 하라.

### 4. WORKSTATE 직접 쓰기 경로를 제거한다

지금은 실행자가 WORKSTATE를 손으로 고친다. **그 경로를 없애라.** 앞으로는 `state-transition`만이 WORKSTATE를 쓴다.

- ❌ **범용 transition engine / 상태 머신 / TransitionRecord 전체** — v9 Phase 2 예약분이다. **Phase 0용 최소 단일 writer만.**
- ❌ **새 canonical ID 체계** — v9를 쓴다.
- ❌ **`server/Harness/**` 접촉** — 코덱스 배타 영역(ADR-002). `handoff-integrity`의 `blockers[]` 검사 확장은 **코덱스 몫이다.**
- ❌ **LLM 기반 상태 추론** — 보고서를 읽고 "끝난 것 같다"고 판단하는 코드 금지.
- ❌ 자동 gate 승인 · 자동 push · 자동 Phase 진입.

## 필수 반증 시험 (사본에서. **원본 WORKSTATE를 망가뜨리지 마라**)

1. stale `expected-sha256` → **WORKSTATE가 바뀌지 않는다**
2. 같은 `transition-id` 두 번 → **두 번째는 상태를 바꾸지 않는다**
3. verification verdict 없이 `completed` → **실패**
4. 사람 결정 없이 Phase 전이 → **실패**
5. canonical 필드에 `P0-07`·`LEDGER-04` → **실패**
6. `blocked`인데 `blockers` 빈 배열 → **실패**
7. `verifying`인데 `nextActions` 빈 배열 → **실패**
8. `projection` 실패 시 → **자동으로 다음 작업을 발사하지 않는다**
9. **정상 전이 후 WORKSTATE와 L0가 같은 상태를 표현한다**

**통과를 믿기 전에 실패시킬 수 있음을 먼저 증명해라.**
