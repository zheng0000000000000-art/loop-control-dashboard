너는 05H-R1 실행자다. 저장소 루트는 C:\Users\1\Documents\Local-First Workflow Dashboard 다.
**현재 브랜치는 `wp/state-integrity`다(통합 branch). 브랜치를 바꾸지 마라.**

## 먼저 알아야 할 것 — 이건 네 전임자의 잘못이 아니다

05H는 반려됐다. 그런데 **반려의 절반은 검수자(나) 책임이다.**

05H 프롬프트의 판정선 2번이 **"at-rest 현재 저장소 → exit 0"**이라고 명령했다.
그런데 지시서 §5 규칙 2(`stateIdSet ⊆ successfulLogIdSet`)를 지키면 **at-rest는 exit 0이 될 수 없다.**
저장소에 실제 오염이 1건 있기 때문이다(아래 §왜).
**두 요구가 모순이었다.** 전임자는 규칙을 완화해 at-rest를 초록으로 만들었고, 그것을 잔여위험 #3으로 **자진 신고했다.**
신고는 옳았다. 판정선이 틀렸다.

**이 프롬프트는 그 모순을 제거한다. at-rest는 이제 exit 1이 정상이다.**

## 왜 (실측 — 검수자가 직접 쟀다)

**저장소 상태에 실제 오염이 1건 있다.**

- `docs/handoff/WORKSTATE.applier-log.jsonl:20`
  → `{"transitionId":"DI0004-BLOCKED-CODEX","result":"handoff-integrity-failed exit=1","exitCode":1,...}`
- `docs/handoff/WORKSTATE.json:384`
  → **같은 id가 `appliedTransitions`에 들어 있다**

**적용이 실패했다고 자기 로그가 말하는 전이가 상태에 남아 있다.**
이것은 WP-STATE-INTEGRITY 근본결함 #2(post-apply 실패 시 rollback 없음)의 **실측 흔적**이고,
규칙 2는 **바로 그것을 잡으라고** 있는 규칙이다.

검수자 반증 시험 F로 현 구현이 이걸 놓치는 것을 실증했다:
state=[SNAP-001, FAILED-BUT-APPLIED] / log에 FAILED-BUT-APPLIED가 exitCode=1로만 존재
→ **기대 exit 1인데 실제 exit 0, failures=[]**

## ★ at-rest는 exit 1이 정상이다 — 초록으로 만들지 마라

**이 작업이 끝나면 `handoff-integrity` at-rest는 exit 1을 낸다. 그것이 올바른 결과다.**

- 실패는 `state-transition-not-logged` **정확히 1건**(`DI0004-BLOCKED-CODEX`)이어야 한다.
- **2건 이상이면 보고하라 — 오염이 더 있다는 뜻이다.** 그건 새 정보다.
- **at-rest를 exit 0으로 만들려고 규칙을 완화하거나 예외·플래그·allowlist를 추가하지 마라. 그게 05H가 반려된 이유다.**
- 빨간불을 끄는 것은 하네스가 아니라 **06C-2 trust-origin 부트스트랩 + 사람 결재**다. 이미 HUMAN-INBOX에 올라가 있다.
- **`handoff-integrity`를 부르는 다른 게이트가 잠길 수 있다. 잠기면 잠긴 대로 보고하라 — 풀려고 하지 마라.**
  어디가 잠기는지가 06C-2가 알아야 할 정보다.

## 절대 규칙

- **`WORKSTATE.json`·`WORKSTATE.applier-log.jsonl` 수정 금지.** append-only 증거다. **`DI0004-BLOCKED-CODEX`를 지우지 마라.**
- **`server/StateApplierCli.cs` 무접촉.** 06C-1의 영역이다. 읽기만.
- **`server/Harness/DiCompletionCheckCli.cs`·`ClaimCheckCli.cs` 무접촉** — `CODEX-GATE-04`의 영역이다.
- **내부 checker를 `HarnessRegistry`에 등재하지 마라**(원 지시서 §2).
- **`PendingTransitionId` 면제는 그대로 유지한다.** 그건 정당한 예외다(state에 1회 · log엔 없음).
- **`allLogIdSet` 자체를 지우지 마라** — Metrics·malformed 판정에 쓰이면 그대로 둔다. **규칙 2에서만 안 쓴다.**
- **원 지시서 §1~§9의 다른 규칙은 손대지 마라.** 이번 작업은 규칙 2와 fixture-f 둘뿐이다.
- **git commit/push 금지. 브랜치 변경 금지. 결재·반입·발사 금지.**
- **`measure`는 `dotnet run --project server -c Release -- measure dev-pack`으로.** exe 직접 호출은 부모 폴더를 봐서 exit 2가 난다(실측).
- 함수마다 한국어 기능 주석 1줄. 빌드는 `-c Release`.
- **⚠️ `di-completion-check`는 하위 검사를 Debug 바이너리로 돌린다**(실측 결함, `CODEX-GATE-04`가 고친다). 코드를 고쳤으면 **`dotnet build server`(Debug)도 한 번** 돌려야 게이트가 새 코드를 본다.

## 판정선 (지시서 §4)

1. `dotnet build server -c Release` → **0**
2. **fixture-f → exit 1 `state-transition-not-logged`** ← **★ 이번 작업의 핵심**
3. fixture A → 1 · B → 1 · C → 0 · D → 1 · E → 1 · malformed → 2  (**회귀 없음**)
4. CLI `--pending-transition` → 1 `pending-not-allowed-on-cli`
5. 내부 checker: state에 X 1회 + log에 X 없음 + **Pending=X → PASS(면제)** / **Pending 없음 → FAIL** `state-transition-not-logged`
6. **at-rest 현재 저장소 → exit 1** · failures에 `state-transition-not-logged` **정확히 1건**
   ← **★ 이것이 정상이다. 05H 프롬프트의 "exit 0"은 틀린 요구였다.**
7. `dotnet run --project server -c Release -- measure dev-pack` → **0**
