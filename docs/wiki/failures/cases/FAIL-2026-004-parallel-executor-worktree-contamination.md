# FAIL-2026-004 — 병렬 헤드리스 실행자가 작업트리를 오염시킴

- 상태: 완화됨
- 최초 발생일: 2026-07-10
- 최근 발생일: 2026-07-10
- 관련 DI: v9 다중모델 FILE-CLAIMS 설계, 모바일 토큰 수정, CLI 분리 리팩토링
- 구성요소: executor-orchestration
- failureClass: concurrency, design_learning
- 심각도: high

## 발생 상황

여러 헤드리스 실행자(sonnet)를 짧은 간격으로 동시에 발사했다. 모바일 토큰 수정과 CLI 분리 리팩토링 같은 작업이 같은 저장소 작업트리를 공유했고, 파일 소유권 예약(`FILE-CLAIMS`)은 도입되지 않은 상태였다.

## 관찰된 증상과 영향

리팩토링 실행자는 `server/`만 만져야 했지만, 검수 시점에 `dashboard/app.js`에서 어느 실행자가 만들었는지 추적할 수 없는 변경이 함께 발견됐다. 해당 변경은 `window.prompt`에서 `promptModal` 개선으로 이어졌고, 결과물은 우연히 `promptModal` 함수·lang·DOM까지 완결되어 유효했다. 그러나 변경의 출처와 의도를 사후에 확정할 수 없어 통제 실패로 기록한다.

## 발생 이유(직접·근본·기여, 미확정은 가설 표시)

- 직접 원인: 동시 실행자들이 파일 단위 소유권 없이 같은 작업트리를 썼다.
- 근본 원인: v9 다중모델 설계의 `FILE-CLAIMS.json` 파일 소유권 예약이 미도입인 상태에서 병렬 작업을 수행했다.
- 기여 요인: 실행 시작 응답인 `launched`를 실행 성공으로 간주해 실제로 몇 개의 실행자가 작업 중인지 관측이 부정확했다.
- 미확정: `dashboard/app.js` 변경 주체는 확정되지 않았다.
- 가설: `dashboard/app.js` 변경은 이전 토큰 관련 실행자가 만든 변경일 가능성이 있다.

## 검토한 해결 방법

- 헤드리스 실행자를 순차로만 실행한다.
- `FILE-CLAIMS`로 파일 소유권을 예약한다.
- 실행자별 전용 작업 사본 또는 worktree를 분리한다.

## 선택한 해결 방법

현재 해결은 헤드리스 실행자 순차 실행 원칙 채택이다. 동시 발사는 금지한다. `FILE-CLAIMS`와 worktree 분리는 향후 병렬 필요성이 다시 커질 때 도입한다. 도입 트리거는 v9 §0-A.6 기준인 동시 실행자 3인 이상 또는 동일 파일 충돌 2회 이상이다.

## 판단 기준

추적 가능성을 처리량보다 우선한다. 결과가 유효하더라도 출처를 알 수 없는 변경은 통제 실패로 간주한다.

## 검증 결과

검수 시점에 `dashboard/app.js`의 출처 불명 변경이 발견됐고, 사후 검토 결과 변경 자체는 함수·언어 리소스·DOM 연결이 완결되어 커밋 가능한 상태였다. 그러나 어느 실행자가 어떤 의도로 만들었는지는 확정하지 못했다.

## 재발 방지

- 헤드리스 실행자는 순차 실행한다.
- 실행 전후 `git status`로 작업 영역을 확인한다.
- 지시서마다 실행자별 영역 격리를 명시한다.
- 병렬 재개 전에는 `FILE-CLAIMS` 또는 작업 사본 분리를 먼저 도입한다.

## 후속 작업과 잔여 위험

- 순차 실행은 처리량을 낮춘다.
- 병렬을 재개하면서 `FILE-CLAIMS`가 없으면 같은 문제가 재발할 수 있다.
- `dashboard/app.js` 변경 주체는 미확정으로 남는다. 현재 가설은 이전 토큰 관련 실행자다.

## 발생 이력

- 2026-07-10: 여러 헤드리스 실행자를 짧은 간격으로 동시 발사.
- 2026-07-10: 리팩토링 검수 중 `dashboard/app.js` 출처 불명 변경 발견.
- 2026-07-10: 순차 실행 원칙을 현재 완화책으로 채택.
