# 스킬: HS-GATE 승격 심사 (하네스·스킬 후보 점수화)

트리거: `dotnet run --project server -- hs-scan` 이 **exit 1**을 반환했을 때. 재량 아님 — **의무**.
(하네스가 아직 없으면: `docs/wiki/failures/index.md`에서 같은 failureClass가 2회 이상인지 수동 확인.)

## 이 스킬이 왜 있나
승격 절차는 KNOWLEDGE-PROMOTION.md에 이미 문서로 있었지만 **한 번도 실행되지 않았다** — HS-CANDIDATES.md가 2026-07-11까지 존재하지 않았던 것이 증거다. 트리거가 "해당하면 판정하라"는 재량이었기 때문이다. 이제 탐지는 `hs-scan`(프로그램)이 하고, **점수화만 여기서(LLM) 한다.**

## 절차

1. **탐지 결과 수용**: `hs-scan` JSON의 `candidates[]`를 그대로 받는다. 후보를 임의로 기각하지 말 것 — 점수로 기각한다.
2. **후보별 점수화** (v9 §0.4, 각 0~2, 총 0~12):

   | 항목 | 0 | 1 | 2 |
   | --- | --- | --- | --- |
   | 반복성 | 1회성 | 2회 | 3회+ 또는 매 회차 반복 |
   | 결정가능성 | 사람 해석 필요 | 부분 자동 | 완전 기계 판정(PASS/FAIL 이분) |
   | 장애주입 | 재현 불가 | 어려움 | 인위적 재현 자명 |
   | 격리 | 부작용 큼 | 일부 상태 변경 | 읽기 전용·부작용 0 |
   | 관찰성 | 출력 불명확 | 결과만 | 왜 실패했는지 분해 출력 |
   | 유지가치 | 일회성 | 가끔 유용 | 게이트·루프가 직접 의존 |

3. **판정**: `0~4 부적합 / 5~7 보류·기존확장 / 8~10 기존확장·기한부 / 11~12 즉시제작`.
   점수와 무관한 차단(보안·정책 미확정)은 보류로 내린다.
4. **유형 결정**: 기계 판정 가능하면 **하네스(CLI)**, 사람·LLM 절차면 **스킬**, 나머지는 위키. (하네스 > 스킬 > 문서)
5. **기록**: `docs/handoff/HS-CANDIDATES.md`에 append — 후보·6항목 점수·총점·판정·**근거 데이터(FAIL ID)**·제안 산출물.
   메타 두 줄을 갱신한다(hs-scan이 읽는다):
   - `lastGate: YYYY-MM-DD HH:MM`
   - `judgedClasses: <이번까지 심사한 failureClass 목록, 쉼표구분>`  ← 같은 계열 중복 제기 방지
6. **승격 요청**: `즉시제작`·`기한부`는 검수자에게 올린다.
   - 하네스 → SONNET-QUEUE에 "하네스 CLI 구현" 지시서 등재 요청(server/, CliRouter 등록, PASS/FAIL 기계 판정, **장애주입 회귀 테스트 필수**).
   - 스킬 → `skills/` 초안.
   - **큐 등재·발사는 검수자·사람의 몫.** 이 스킬은 심사만 한다.

## 하지 않는 것
- 결재·반입·기준 변경 대행. 발사. git push. (전부 사람 게이트)
- 후보를 재량으로 묵살하기. 점수가 낮으면 낮다고 기록한다 — **기각도 기록이 남아야 다음 반복에서 근거가 된다.**

## 좋은 심사의 예 (실측)
HS-01 `gate-clean`(12/12 즉시제작). 근거: FAIL-2026-005(실행 판정을 StartTime 프록시에 의존 → 오판)와 FAIL-2026-010(트리 판정을 줄바꿈 표현에 의존 → 교착)이 **같은 실패계열 `unnormalized_gate` 2회**. 두 건이 각각 있을 땐 그냥 사고였지만, **반복이 드러나는 순간 "게이트 판정을 표현·프록시에 의존시키지 말라"는 규칙**이 되고, 그 규칙은 CLI로 고정 가능하다 → 하네스.
이것이 이 스킬이 찾는 패턴이다: **개별 사고 2건 → 규칙 1개 → 실행 가능한 검사 1개.**
