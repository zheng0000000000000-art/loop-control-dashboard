# FAIL-2026-010 — 줄바꿈 표현 차이가 발사 게이트를 영구 잠금(큐 정지)

- 상태: 해결됨
- 최초 발생일: 2026-07-09 (mtime 기준 잠복 시작)
- 최근 발생일: 2026-07-11 (발견·해결)
- 관련 DI: SONNET-QUEUE 발사규칙 ①, ORCH-01(관측 스캐폴드), HARNESS-01(gate-clean)
- 구성요소: executor-orchestration, gate-evaluation
- failureClass: unnormalized_gate, config_side_effect, design_learning
- 심각도: high

## 발생 상황

조율자(5분 루프)는 SONNET-QUEUE 자동발사 규칙 ①로 `git status --porcelain -- server`가 비어야(=server/ clean) 다음 지시서를 발사한다. 2026-07-11, 큐가 #3(FEAT-02)에서 넘어가지 않는다는 관찰로 조사에 착수했다.

## 관찰된 증상과 영향

- 조율자가 매 회차 **두 가지를 동시에** 보고: "server/docs 변경 없어 커밋할 대상 없음" + "다음 항목(FEAT-02) 발사 대기".
- 큐가 #3에서 무기한 정지. 구현 루프가 사람이 직접 발사할 때만 진행됨(FIX-01·FIX-02가 그렇게 돌았다).
- 두 보고는 모순처럼 보이지만 실은 같은 원인의 두 얼굴이었다. 이 모순이 리포트에 나란히 찍히는데도 수 회차 동안 아무도 교차검증하지 않았다.

## 발생 이유(직접·근본·기여, 미확정은 가설 표시)

- **직접**: 어떤 도구가 `server/` 5개 파일(DevPackMeasures·OllamaExecutor·OutboxManager·Program·Storage)을 LF→CRLF로 재작성했다. **실내용 변경은 0** — CR만 제거하면 HEAD 블롭과 SHA가 5개 전부 완전 일치한다. 바이트 증가분이 줄 수와 정확히 일치(예: 891줄 → +891바이트).
- **근본**: 저장소에 `.gitattributes`가 없고 `core.autocrlf`도 unset이었다. 줄끝을 정규화하는 장치가 **전무**하여 git이 표현만 다른 파일을 영구히 "수정됨"으로 인식했다.
- **근본(설계)**: 발사 게이트가 "트리가 깨끗한가"라는 **의미론적 질문**을 `git status`라는 **표현 의존 프록시**로 판정했다. 표현(줄바꿈)이 흔들리자 게이트가 영구 거짓이 됐다.
- **기여**: 조율자는 내용 기준으로는 정직하게 "커밋할 것 없음"이라 판단했다. 그래서 **커밋할 것도 없고 dirty라서 발사도 못 하는 교착**이 완성됐다 — 어느 쪽도 스스로를 풀 수 없다.
- (가설) CRLF 재작성 주체는 Windows에서 파일을 되쓰는 에이전트 편집 도구로 추정. mtime이 07-09·07-10·07-11로 흩어져 있어 특정 1회 사건이 아니라 **파일을 건드릴 때마다** 발생한 것으로 보인다. 주체 미확정.

## 검토한 해결 방법

1. `git checkout -- server/`로 되돌리기 — 일시적. 다음에 도구가 파일을 건드리면 즉시 재발.
2. 조율자 발사규칙에서 조건 ①을 삭제 — 게이트 약화. 진행 중 작업 위에 발사하는 FAIL-004 재발 위험. 기각.
3. `.gitattributes`로 줄끝 정규화 + 게이트 판정을 **정규화 내용 해시**로 이전 — 표현 차이를 게이트에서 구조적으로 제거.

## 선택한 해결 방법

3안. 두 층으로 나눴다.

- **즉시(저장소 위생)**: `.gitattributes` 신설(`* text=auto eol=lf`, `*.cs text eol=lf`) + `git add --renormalize .`. server/ clean 회복, 내용 손실 0. 커밋 `8545903`.
- **구조(게이트 정정)**: 게이트 판정을 raw `git status`가 아니라 **정규화된 내용 해시 비교**로 하는 하네스 `gate-clean`을 승격(HARNESS-01). 표현만 다른 파일(CRLF/BOM)과 실내용이 바뀐 파일을 **분해해서** 리포트하고, 발사 게이트는 후자만 본다.

## 판단 기준

- 게이트가 답해야 할 질문은 "누가 server/에서 작업 중인가"이지 "바이트가 다른가"가 아니다. 후자는 전자의 불안정한 프록시다.
- 위생 수정(.gitattributes)만으로는 부족하다 — 정규화되지 않은 다른 표현(BOM, 후행 공백, 권한 비트)이 언제든 같은 교착을 만든다. 게이트 자체가 표현에 둔감해져야 한다.

## 검증 결과

- 5개 파일 전부 CR 제거 후 HEAD와 SHA 일치 확인 → checkout/정규화가 무손실임을 사전 증명.
- `.gitattributes` + `--renormalize` 적용 후 `git status --porcelain -- server` = 빈값(clean).
- 발사 조건 4개 재실측: ①clean ✅ ②실행 중 sonnet 없음 ✅ ③진행 항목 0 ✅ ④다음 대기(FEAT-02) 존재 ✅ → 교착 해소.
- `git diff --cached --ignore-all-space` = 0건 → 정규화가 루프의 미커밋 작업을 삼키지 않았음 확인.

## 재발 방지

- **게이트 판정은 정규화된 내용 해시로 한다.** 줄바꿈·BOM·후행공백 같은 표현 차이는 게이트 입력에서 제거한다(HARNESS-01 `gate-clean`).
- 저장소는 `.gitattributes`로 줄끝을 고정한다(LF 기준).
- **리포트 교차검증**: 조율자가 "커밋할 것 없음"과 "발사 대기"를 동시에 보고하면 **둘이 같은 원인일 수 있음**을 의심하는 검사를 넣는다. 모순처럼 보이는 두 진술이 나란히 찍히는 것 자체가 신호였다.
- ORCH 오케스트레이터의 clean 판정은 `git status` 문자열이 아니라 `gate-clean` 하네스를 호출한다.

## 후속 작업과 잔여 위험

- HARNESS-01(`gate-clean`) 구현 대기(SONNET-QUEUE #6). 그전까지 게이트는 `.gitattributes` 위생에만 의존한다 — 정규화되지 않은 다른 표현이 나타나면 재발 가능.
- CRLF 재작성 주체 미확정. 주체를 특정하지 못하면 `.gitattributes`가 없는 다른 저장소·경로에서 같은 문제가 난다.
- server/가 clean해져 발사 조건이 전부 열렸다. 자동발사가 켜져 있다면 다음 회차에 FEAT-02가 무인 발사될 수 있다 — 발사는 사람 게이트라는 전제가 유지되는지 확인 필요.
