# ADR-008 — 모델 출력 오류를 **정규화하되 감추지 않는다** (metricId 대소문자)

- 상태: **승인됨 (사람 choi, 2026-07-12 03:5x)**
- 결정 주체: 사람. 선택지 3개를 검수자가 제시하고 사람이 골랐다.
- 산출물: `LEDGER-04` 지시서 → 발사(PID 34288)
- 근거 실측: `outputs/reviewer-log.md` 2026-07-12 01:4x 항목 · `docs/verification/ledger03-fallback-observability.md`

## 1. 상황 (실측 — 추측 아님)

토큰 계측(LEDGER-01/02)을 켜자마자 드러났다:

```
00:57:50  review.tier1_completed   provider=ollama       in=1541 out=147   ← ollama 호출됨
00:57:56  proposal.generated       provider=rule-engine  in=0    out=0     ← 6초 뒤, 산출물은 rule-engine
```

`server/OllamaExecutor.cs`의 `ParseNoteResponse`가 `metricId == expectedMetricId`로 **대소문자 완전일치** 대조를 한다. `qwen3:8b`는 `functionsWithoutComment`를 **`functionsWithOutComment`(대문자 O)**로 반환한다 → 거부 → `null` → **rule-engine 폴백.**

**그리고 그 폴백이 완전히 조용했다.** run-log에 `fallback`·`warn`·`error` 이벤트가 **0건**이었다(LEDGER-03이 이 침묵을 걷어냈다).

**결과: 우리는 로컬 LLM을 쓴다고 믿으면서 rule-engine 출력을 받아왔다.** 마지막 정상 ollama 제안은 2026-07-11 23:40. **회귀가 아니라 원래 그랬고, 아무도 몰랐다.**

## 2. 선택지

**(A) 파서를 대소문자 무시로 완화한다.**
- 장점: 즉시 복구. 한 줄이면 된다.
- 단점: **모델 출력 오류를 코드가 흡수해 감춘다.** 다음에 모델이 **다른 방식으로** 틀리면 **또 조용해진다.** 우리는 방금 그 침묵 때문에 몇 시간을 잃었다.

**(B) 정규화 + 계속 기록 (선택)**
- 1차는 지금처럼 엄격 대조 → 통과하면 아무 이벤트도 남기지 않는다.
- 2차로 **대소문자 무시 대조만** 재시도 → 통과하면 **수용하되 `proposal.metricid_normalized`(`level: warn`)를 계속 남긴다.**
- 장점: **ollama는 살리고, 모델 품질 저하는 계속 보인다.** 관측을 끄지 않는 복구.
- 단점: 이벤트가 하나 늘고, 코드가 조금 복잡해진다.

**(C) 프롬프트를 고쳐 모델이 정확한 metricId를 뱉게 한다.**
- 장점: 근본 교정.
- 단점: **모델 의존적.** 모델을 바꾸면 다시 깨진다. 그리고 프롬프트로 강제하는 것은 이 프로젝트가 계속 실패해온 방식이다 — **"프롬프트로 시키지 말고 코드로 강제하라"**(INTENT-DIGEST).

## 3. 선택: **(B)**

## 4. 판단 기준

1. **복구하되 눈을 감지 않는다.** (A)는 문제를 없애는 게 아니라 **보이지 않게** 만든다. 오늘의 병은 "고장"이 아니라 **"고장이 안 보였던 것"**이다.
2. **관대함의 범위를 코드로 좁힌다.** 대소문자 **한 가지로만** 한정한다. 공백 제거·유사도 매칭·부분 일치·"metricId 목록 중 가장 비슷한 것 고르기"는 **전부 금지**다 — 그건 검증이 아니라 **추측**이고, 추측이 코드에 들어가면 다시는 안 보인다.
3. **`warn`을 `info`로 낮추지 않는다.** 낮추는 순간 이 결정의 의미가 사라진다. 모델 품질 저하는 **시끄러워야 한다.**
4. **`role`은 `runtime` 유지, `cost` 스키마 불변.** 스키마 변경은 별도 결재 사항이다.

## 5. 이 결정이 여는 것

**LEDGER-02의 토큰 배선은 아직 한 번도 실행된 적이 없다** — ollama 제안 경로가 폴백으로 죽어 있었기 때문이다. **(B)로 그 경로가 살아나면 `proposal.generated`에 `provider: ollama` + `cost.inputTokens > 0`이 처음 찍힌다.** 그것이 LEDGER-02의 첫 검증이다. **두 작업은 여기서 만난다.**

## 6. 되돌림 조건

- 정규화 이벤트(`proposal.metricid_normalized`)가 **거의 모든 호출에서** 발생하면, 그건 모델이 규약을 아예 못 지킨다는 뜻이다 → **(C) 프롬프트 교정** 또는 모델 교체를 검토한다.
- 반대로 이벤트가 **0건**이면 모델이 정상화된 것이므로 정규화 분기를 제거해도 된다. **데이터가 판단한다.**
- 되돌리는 법: `OllamaExecutor.ParseNoteResponse`의 2차(대소문자 무시) 분기를 제거하면 엄격 대조로 복귀한다. `proposal.fallback`(LEDGER-03) 이벤트는 그대로 남는다.

## 7. 관련 실패 사례

- **FAIL-2026-012**(프록시로 원인 단정) — 이 건은 그 반대다. 침묵을 **관측으로 바꾼 뒤에야** 원인을 실체로 지목했다.
- 침묵의 대가: 마지막 정상 ollama 제안 **23:40** → 발견 **다음날 01:1x**. **토큰을 재지 않았으면 영영 몰랐다**(ADR-006이 옳았다는 증거).
