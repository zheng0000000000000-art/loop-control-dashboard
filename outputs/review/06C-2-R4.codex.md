### 지표는 통과하지만 목적을 배신하는 경로
- R3 우회: stderr에 `trusted-human-receipt-required` 문자열만 포함시키기 / reason 자체가 다른데도 PASS 되는 문제 / 지금 구현에 실재하지 않음. `server/TrustOriginCli.cs:286-297`이 각 stderr 라인을 JSON으로 파싱하고 `reason` 또는 `code` 필드가 정확히 일치해야만 `reasonMatched=true`가 된다. 반증 입력 `{"code":"other","detail":"trusted-human-receipt-required"}`는 현재 false가 된다.

- exit code만 맞추기 / high-risk가 아닌 실패도 PASS 되는 문제 / 지금 구현에 실재하지 않음. `server/TrustOriginCli.cs:254-255`가 `exit == 1`과 `reasonMatched == true`를 같이 요구한다. 반증 입력: exit 1 + stderr `{"reason":"state-corrupted-preapply"}`.

- high-risk self-test가 tmpRoot fixture만 검증하는 척하기 / canonical WORKSTATE 의존성을 숨기는 문제 / 의존성은 실재하지만 보고되어 계약 위반은 아님. `server/TrustOriginCli.cs:283`이 `StateApplierCli.Run(...)`을 호출하고, `server/StateApplierCli.cs:226`이 repo root WORKSTATE를 읽는다. 다만 `server/StateApplierCli.cs:241-244`에서 request/candidate 읽기 전 fail-closed하므로, R4 문서도 이 의존성을 `docs/verification/06c2-r4-trust-origin-review-contract.md:83-89`에 명시했다.

- hooks fixture가 사용자 HOME 설정에 영향받기 / tmpRoot만의 no-hooks 검증이 아니게 됨 / 부분적으로 실재함. `server/TrustOriginCli.cs:237-246`은 tmpRoot뿐 아니라 HOME `.claude/settings*.json`도 검사한다. 현재 읽은 HOME 설정에는 `hooks` 필드가 없어 reported no-hooks 결과와 충돌하지 않지만, HOME에 `{"hooks":{}}`가 있으면 `launcher-settings-no-hooks` fixture는 false가 된다. 이는 false PASS보다는 환경 의존 테스트 실패 위험이다.

### 테스트가 못 잡는 것
- `code` 전용 출력 경로. 현재 실제 `StateApplierCli.Fail`은 `reason` 필드를 낸다(`server/StateApplierCli.cs:917-925`). matcher는 `code`도 처리하지만 self-test는 `{"code":"trusted-human-receipt-required"}`만 있는 stderr를 직접 주입하지 않는다.

- HOME 설정 오염. 반증 입력: `%USERPROFILE%\.claude\settings.json`에 `{"hooks":{}}`가 있고 tmpRoot `.claude/settings.json`은 `{}`인 상태. `launcher-settings-no-hooks`는 fixture 기준으로는 true여야 하지만 현재 구현은 HOME까지 읽어 false가 된다.

- settings 루트가 객체가 아닌 valid JSON인 경우. 반증 입력: `.claude/settings.json`이 `[]` 또는 `"x"`. 현재 `server/TrustOriginCli.cs:228-229`는 `JsonObject`가 아니면 hooks 없음처럼 true를 반환한다. R4 계약은 malformed와 hooks 필드 존재만 명시했으므로 위반으로 단정하지 않았다.
