# 스킬: 지시서 작성 (지시자)

버전: 1 | 도메인: common | 트리거: 항상 | 대상: **지시서를 쓰거나 고칠 때.** (지시서를 **읽고 구현**하는 쪽은 `directive-writing.md` — 이름과 달리 그쪽이 '해석' 스킬이다)

> **왜 이 스킬이 생겼나 (실측)**: `GATE-TRUTH-01` 지시서가 **네 번 연속 틀렸다**(초판·R1·R2·R3). 매번 사람 검수가 잡았다.
> 네 번의 결함이 **같은 모양**이었다. 사람이 매번 다시 잡는 것은 절차가 아니다. **여기 적어서 다음 사람이 안 반복하게 한다.**
> 기계가 잡을 수 있는 것은 `directive-lint` 하네스가 잡는다(`DLINT-01`). **이 스킬은 기계가 못 잡는 것만 다룬다.**

---

## 0. 착수 전 — **재발명 금지 3종 검색** (하나라도 건너뛰면 중복 제작한다)

지시서를 쓰기 시작하기 전에 **반드시** 셋 다 한다.

1. **이 기능이 이미 있는가?** 고치려는 대상 파일을 열어 grep한다.
   - *실측*: `GATE-TRUTH-01` R2가 reconciliation 판정을 새로 설계했는데, **`TrustOriginCli.cs:293~295`에 exact binding이 이미 있었다**(`EntrySetHash`·`FailureSetHash`·`ReconciliationReportHash`). 넣었으면 **판사가 둘**이 될 뻔했다.
2. **다른 미완 지시서가 같은 파일·같은 기능을 이미 소유하는가?** `docs/handoff/queue/` 전체를 grep한다.
   - *실측*: 초판이 `--emit-hashes`를 요구했는데 **`CODEX-GATE-04` §5-4에 같은 문장으로 이미 있었다.**
   - allowlist가 겹치면 **`## 선행` 절에 그 diId를 적는다.** 안 적으면 `directive-lint`가 exit 1로 막는다.
3. **"이미 있다"고 남이 적어둔 말을 믿지 말고 열어봐라.**
   - *실측*: `CODEX-GATE-04` §5-3이 *"지시서 lint는 `LaunchCheckCli`의 확장일 수 있다"*고 추정했다. **열어보니 발사 프롬프트 전송 증거의 해시를 검사하는 것이었다 — 무관했다.**

## ★ 1. 게이트 분류 규칙 — **이 스킬에서 제일 중요한 절**

**exit code를 기대값으로 판정하는 게이트 배열에 검사를 넣을 때, 셋을 전부 만족해야 한다.**

```
C1  exit code가 "정확성"을 뜻한다           ← "지금 이런 상태다"를 뜻하면 안 된다
C2  wall-clock과 무관하게 같은 exit를 낸다   ← replay/재현 대조의 전제조건
C3  다른 관찰과 판정 대상이 겹치지 않는다     ← 판사가 둘이 되면 안 된다
```

**하나라도 어기면 게이트가 아니라 `exact-binding 관찰`로 간다** — 실체(집합·해시)를 결속하되 exit code로 판정하지 않는다.

> **⚠ `expectedExit != 0`이 필요하다고 느끼는 순간을 경보로 여겨라.**
> 그건 십중팔구 **그 검사가 "상태"를 보고하고 있다**는 뜻이다. 기대값을 1로 적지 말고, **배열에서 빼라.**

**실측 사례 — 같은 실패를 네 번 했다:**

| 지시서 | 넣은 검사 | `expectedExit`가 인코딩한 것 | 터졌으면 |
| --- | --- | --- | --- |
| R1 | `gate-clean` (POST-EXECUTOR) | "트리는 항상 더럽다" | 깨끗한 worktree에서 영구 FAIL |
| R2 | `handoff-integrity` = 0 | "baseline은 항상 정합" | legacy-gap baseline에서 영구 차단 |
| R3 | `hs-scan` = 1 | "승격 후보가 항상 있다" | **`DateTime.Now` 의존 — replay가 깨진다** |
| R3 | `recovery inspect` = 0 | "pending이 항상 0" | quarantine 1건이면 뒤집힘 |

**`hs-scan`은 특히 악질이다** — `HsScanCli`가 `daysSince >= 1.0`으로 신호를 만들고 `return triggered ? 1 : 0`한다.
**시각이 지나면 같은 baseline이 다른 exit를 낸다.** 관찰 시점과 검증 시점이 다른 모든 설계가 여기서 깨진다.

**지시서에 이 표를 채워 넣어라.** 배열에 넣은 것·뺀 것 **전부**, C1/C2/C3 판정과 함께. 못 채우면 넣지 마라.

## 2. 구멍을 막는 지시서는 **"고치기 전엔 뚫린다"를 먼저 시킨다**

완료 기준에 **★ 재현 단계**를 넣는다. 순서는 고정이다:

```
① 고치기 전 코드로 구멍을 실제로 통과시켜 보여라   ← 이게 없으면 "막았다"가 성립하지 않는다
② 고친 뒤 같은 조건에서 막히는지 보여라
```

- **재현은 `$TEMP`의 폐기 가능한 저장소에서.** 저장소 안에서 시험하다 **`docs/wiki` 42파일을 지운 사고**가 있었다.
- **프로덕션에 부작용을 남기는 재현은 금지.** 보고에 `PRODUCTION_*_CREATED = false`처럼 **기계가 읽을 수 있는 줄**로 못 박게 한다.

### ★ 2-1. **네가 설계한 방어를 네 손으로 뚫어봐라**

*실측*: 초판이 *"`declare`가 `verdict` 문자열 대신 `actualExit == expectedExit`로 재판정하라"*고 썼다.
**evidence 파일을 사람이 고칠 수 있으면 `actualExit`도 기대값으로 타이핑하면 그만이다.** 이름만 바뀐 같은 구멍이었다.

**자문하라: "이 파일을 쓰는 사람이 악의적이면 어디를 타이핑하는가?"**
답이 "내가 판정에 쓰는 필드"면 **그 설계는 틀렸다.** 판정은 **파일이 아니라 재실행**에 근거해야 한다.

## 3. 지시서 자기 모순 — **내보내기 전 셀프 검사**

*실측*: 초판의 완료 기준 2번이 `declare` 실행을 요구했는데 §2 금지 항목이 `declare` 실행을 금지했다.
**실행자가 둘 중 하나를 어겨야만 완료되는 지시서였다.**

- 완료 기준이 요구하는 명령이 **금지 목록에 있는가?** 있으면 **격리 조건(`$TEMP`·fixture)을 같은 항목 안에 명시**하거나 `## 금지 예외` 절을 만든다.
- **§1에서 "고쳐라"라고 한 파일이 전부 `## 허용 파일 (allowlist)`에 있는가?**
- **`requiredInputs` ∩ `allowlist` = ∅ 인가?** 겹치면 게이트가 **자기 작업에 걸려 넘어진다**(`_header.md`).
  *실측*: `GATE-MANIFEST.json`이 쓰기 대상이 되자 `requiredInputs`에서 빼야 했다.
- **allowlist가 다른 미완 지시서와 겹치는가?** 겹치면 `## 선행`에 diId를 적는다. **동시 발사 금지.**

> 이 넷은 **`directive-lint` 하네스가 기계로 잡는다**(`DLINT-01`). **셀프 검사는 lint를 대신하지 않는다 — lint를 돌려라.**

## 4. 해시·수치·줄번호는 **프로그램이 계산하고, 인용은 열어보고 적는다**

- **`sha256`은 LLM이 적지 않는다.** `context-pack-integrity --emit-hashes`(`CODEX-GATE-04` §5-4) 또는 호스트의 `Get-FileHash`.
  **리눅스 마운트 뷰가 지연된 바이트를 돌려준 실측 사고**가 있다 — **계산하는 쪽과 검사하는 쪽이 같은 프로그램이어야 한다.**
- **`파일.cs:308` 같은 인용은 열어보고 적는다.** `directive-lint`가 파일 존재·줄번호 범위는 잡지만 **내용이 맞는지는 못 잡는다.**
- **"현재 X는 exit 1이다" 같은 상태 주장은 지금 재서 적는다.**
  *실측*: 검수가 *"at-rest `handoff-integrity` exit = 1"*을 전제로 지적했는데, **BC-003 이후 exit 0이었다.** 전제가 낡아도 구조 진단은 맞을 수 있으니 **전제를 정정하고 수정은 받아라** — 그러나 **틀린 전제를 지시서에 박으면 실행자가 그걸 믿는다.**

## 5. 판정이 불편할 때 **기준을 옮기지 마라**

지시서를 쓰다 보면 게이트가 빨간불이라 진도가 안 나간다. 그때 나오는 유혹은 항상 같다:

- `GATE-MANIFEST`의 `expectedExit`를 **실측에 맞춰** 바꾸기
- 검사 하나를 **비교 대상에서 조용히 빼기**
- 기준 파일(`blueprint.json`·`workflow-definition.json`)이나 **측정 코드**를 고치기

**셋 다 "검사를 죽이는 것"이다.** 고쳐야 할 것은 **상태**다.
원칙과 충돌하면 **"기준 변경입니까?"를 사람에게 묻는다.** 기준 변경은 **사람 결재**이고, 바꿨으면 `docs/handoff/BASELINE-CHANGES.md`에 **①주체 ②근거 ③되돌리는 법**을 **append**한다.

**단, 기준을 "추가"하는 것과 "완화"하는 것은 다르다.**
*실측*: `TRUST-ORIGIN-BASELINE` 게이트 세트 **신설**은 정당했다(기존 세트의 전제가 반대라 재사용 불가). **기존 세트의 `expectedExit`를 건드리는 것은 반려다.** 신설이면 **기존 블록 무변경을 `git diff`로 증명**하게 하라.

## 6. 완료 기준은 **exit code로 판정 가능해야 한다**

- "X가 동작한다" → **"이 명령을 돌리면 exit N이 나온다"**로 바꿀 수 없으면 기준이 아니다.
- **출력 문자열·정규식으로 성패를 세지 마라**(`CLAUDE.md`). 판정은 **exit code**로.
- **결정성·격리를 항상 넣는다**: 2회 연속 같은 출력 / 하네스 실행 전후 `git status` 동일.
  **격리 확인은 `git hash-object`로 blob 대조**한다 — `git status`는 프록시다.
- **`--no-build`를 쓰라고 지시하지 마라.** Release 산출물이 낡으면 게이트 넷이 exit 2다(두 세션 연속 실측 사고).
