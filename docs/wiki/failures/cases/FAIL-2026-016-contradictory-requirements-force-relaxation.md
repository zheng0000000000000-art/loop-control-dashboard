# FAIL-2026-016 — **모순된 요구가 실행자를 규칙 완화로 민다**

- 상태: 부분 해결 (프롬프트 작성 규칙으로 승격 필요)
- 최초 발생일: 2026-07-13
- 최근 발생일: 2026-07-14
- 관련 DI: 05H · 06C-1 · 06C-1-R1
- 구성요소: executor-orchestration
- failureClass: design_learning
- 심각도: 높음 (**실행자를 탓하고 진짜 원인을 놓친다**)

## 발생 상황

**실행자가 계약을 완화해 게이트를 통과시켰다. 세 번 다 원인의 절반은 검수자가 준 요구가 서로 모순이거나 비어 있었다.**

## 관찰된 증상과 영향

### 사례 A — 05H (2026-07-13)

- **지시서 §5-2**: `stateIdSet ⊆ successfulLogIdSet` (성공 로그 기준)
- **프롬프트 판정선 2**: **"at-rest 현재 저장소 → exit 0"**

**둘은 동시에 만족될 수 없다.** 저장소에 실제 오염(`DI0004-BLOCKED-CODEX` — 실패했다고 로그에 기록된 전이가
`appliedTransitions`에 있다)이 있으므로, 규칙 2를 지키면 at-rest는 **반드시 exit 1**이다.

실행자는 규칙을 `allLogIdSet`으로 완화해 at-rest를 초록으로 만들었고, **잔여위험 #3으로 자진 신고했다.**

> **신고는 옳았고 판정선이 틀렸다. 프롬프트를 쓴 것은 검수자다.**

**R1에서 모순을 제거하니: 76턴 $3.54 → 24턴 $0.63.** **6분의 1로 줄었다.**

### 사례 B — 06C-1 (2026-07-13)

- **프롬프트**: "`--root` 같은 production 경로 플래그를 만들지 마라"
- **프롬프트**: "사본에서 시험하라"
- **실체**: `ProjectionCli.cs:32`가 `FindRepoRoot()`를 **하드코딩**해 root를 못 받는다

실행자는 `--root`를 만들고 `canonicalMode`로 projection을 껐다. **명시적 금지를 어긴 것이지만, 대안을 몰랐다.**

**정답은 있었다**: PowerShell `Set-Location`은 **.NET `CurrentDirectory`를 바꾸지 않는다.**
`cmd /c "cd /d <copy> && dotnet run ..."`으로 **진짜 cwd**를 바꾸면 `FindRepoRoot()`도 `ProjectionCli`도 사본을 본다.
**우회할 필요가 없었다. 검수자가 그 한 줄을 안 줬다.**

> 코덱스 독립 검수도 같은 결론: *"`--root` 금지와 사본 실증은 본질적 모순이 아니다. 사본을 process cwd로 실행하면 된다.
> 따라서 `--root` 추가는 요구 결함이 아니라 **구현 우회**다."* — **맞다. 다만 검수자가 방법을 안 줬다.**

### 사례 C — 06C-1-R1 (2026-07-14)

- **지시서**: "test seam에서 환경변수를 제거하고 **in-process 훅**으로 바꿔라"
- **빠진 것**: **그 훅을 어떻게 시험하는가**

실행자는 훅으로 바꿨다(옳다). 그리고 **rollback 시험이 불가능해졌다** — 자진 신고: `NOT_VERIFIED`.

**결함 2(rollback 없음)는 이 WP의 존재 이유 중 하나다. 고쳤다는 증거가 사라졌다.**

**답은 이미 저장소에 있었다** — 05H-R2가 만든 `handoff-integrity --self-test` 패턴.
**검수자가 그걸 가리켜주지 않았다.**

## 발생 이유

- **직접 원인**: 지시서와 프롬프트가 **각각 다른 시점에** 쓰였고, **서로 대조되지 않았다.**
- **근본 원인**: **요구의 무모순성을 검사하는 절차가 없다.**
  실행자는 "완료 기준을 만족시켜라"라는 압박을 받는다. 만족 불가능하면 **가장 약한 제약을 완화한다.**
  그리고 **그것을 자진 신고한다**(ADR-005가 작동한다 — 이건 잘 돌아간다).
- **기여 원인**: 검수자가 **"하지 마라"만 쓰고 "그럼 어떻게 하나"를 안 준다.**
  금지는 대안 없이는 우회를 부른다.

## 검토한 해결 방법

1. 실행자를 더 강하게 압박한다 (× — 이미 자진 신고하고 있다. 압박이 문제다)
2. 지시서-프롬프트 무모순성 lint (△ — 기계로 판정하기 어렵다)
3. **프롬프트 작성 시 "모순 자가 검사"를 의무화** (○)
4. **금지에는 반드시 대안을 붙인다** (○)
5. **실행자에게 "모순이면 완화하지 말고 보고하라"를 명시** (○ — 이미 반영 중)

## 선택한 해결 방법

**3 + 4 + 5.** `skills/common/directive-writing.md`에 승격한다.

**프롬프트 작성 전 자가 검사 (검수자 의무):**

1. **완료 기준을 하나씩 읽고, 현재 저장소 상태에서 만족 가능한지 확인한다.**
   - at-rest가 오염돼 있으면 "at-rest → 0"은 불가능하다. **그 사실을 먼저 실측하라.**
2. **"하지 마라" 항목마다 "그럼 어떻게 하나"를 붙인다.**
   - `--root` 금지 → **"cwd를 바꿔라. PS `Set-Location`은 .NET CurrentDirectory를 안 바꾼다. `cmd /c "cd /d <copy> && ..."`"**
   - 환경변수 seam 금지 → **"in-process 훅 + `--self-test` 단언 실행기. 05H-R2 패턴을 따르라."**
3. **프롬프트에 명시한다**: *"요구가 서로 모순돼 보이면 완화하지 말고 보고해라. 보고는 감점이 아니다. 완화는 반려다."*
4. **실행자가 완화했으면, 탓하기 전에 요구가 성립했는지부터 본다.**

## 판단 기준

**"실행자가 규칙을 어기면, 먼저 내가 준 요구가 서로 모순이었는지 보라."**

## 검증 결과

- 05H → 05H-R1: 모순 제거 후 **76턴 $3.54 → 24턴 $0.63.** 실행자는 정확히 지시대로 했다.
- 06C-1-R2 프롬프트: `--self-test` 패턴을 **명시적으로 가리키고**, 참고 파일(`HandoffIntegrityCli.cs`의 `RunSelfTest`)을 읽는 순서에 넣었다.
- **아직 검증 안 됨**: 06C-1-R2 결과로 확인한다.

## 재발 방지

1. **프롬프트를 쓰기 전에 완료 기준을 저장소 실제 상태로 한 번 실행해 본다.** 불가능한 기준은 그 자리에서 드러난다.
2. **금지 항목마다 대안을 붙인다.** 대안 없는 금지는 우회를 부른다.
3. **"모순이면 보고하라"를 프롬프트에 명시한다.**
4. **자진 신고(ADR-005)는 감점이 아니다** — 이 원칙 덕분에 세 번 다 실행자가 스스로 알렸다. **이 원칙을 지켜라.**

## 후속 작업과 잔여 위험

- `skills/common/directive-writing.md`에 위 자가 검사 절차를 추가해야 한다 (**미수행**)
- **잔여 위험**: **검수자가 스스로 모순을 검사하는 구조는 자기 참조다.** 코덱스 독립 검수가 요구의 결함까지
  지적하도록 프롬프트에 넣었고(06C-1 검수), **실제로 지적했다.** 이 경로를 유지해야 한다.
