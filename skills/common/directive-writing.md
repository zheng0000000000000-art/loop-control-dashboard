# 스킬: 지시서 해석

버전: 2 | 도메인: common | 트리거: 항상 | 대상: "작업 지시: ..." 형태의 지시서를 받아 구현 작업을 시작할 때

## 절차

1. **컨텍스트를 읽고 현재 상태를 확인한다.** 지시서가 전제하는 배경(예: "직전 커밋에서 재생성 루프는 동작했으나 ...")이 실제 코드·데이터 상태와 맞는지 먼저 확인한다. 맞지 않으면 지시서의 전제가 틀렸을 수 있으니 진행 전에 짚는다.
2. **불변 제약을 맨 먼저 확인한다.** 불변 제약 전문은 [docs/directives/_header.md](../../docs/directives/_header.md)에 있다. 지시서는 인라인 헤더 대신 "이 지시서는 docs/directives/_header.md의 불변 제약을 따른다" 1줄을 쓴다 — 그 줄을 보면 _header.md를 열어 전문을 읽는다(#11까지의 과거 지시서는 인라인 전문을 담고 있다).
   지시서의 자기완결 원칙은 "외부 링크 금지, 저장소 내 상대경로 참조 허용"이다 — 저장소 밖 URL에 의존하는 지시서는 반려 대상이지만, _header.md 같은 저장소 내 상대경로 참조는 자기완결을 깨지 않는다.
3. **지시서를 알파벳 절(A, B, C...)로 쪼개 각 절을 작업 항목으로 옮긴다.** 절 하나가 보통 파일 하나 또는 기능 하나에 대응한다. 여러 절이 동시에 큰 작업이면 TaskCreate로 추적한다.
4. **"완료 기준" 체크리스트를 검증 가능한 명령으로 번역한다.** "X가 동작한다"는 항상 "이 명령을 실행하면 이 출력이 나온다"로 바꿀 수 있어야 한다 — 바꿀 수 없다면 기준이 모호한 것이니 지시서의 의도를 다시 읽는다.
5. **구현 후 `verification.md` 스킬(아래)대로 실제 실행 검증을 하고, `docs/verification/{주제}.md` 문서를 새로 만든다.** 파일명은 지시서의 핵심 기능을 짧은 kebab-case로.
6. **관례 게이트를 통과시킨 뒤에만 커밋한다.** `dotnet run --project server -- measure <projectId>`를 실행하고 결과를 검증 문서에 기록한다(CLAUDE.md 참조).

## 지켜야 할 것

- 지시서가 명시한 파일 경로·이벤트 이름·필드 이름을 임의로 바꾸지 않는다. 다르게 구현해야 할 기술적 이유가 있으면 검증 문서에 이유를 남긴다.
- "기존 로직 재사용, 중복 구현 금지" 같은 제약이 있으면 실제로 기존 함수를 호출하는지 grep으로 확인한다.
- 지시서에 없는 개선을 끼워 넣지 않는다(스코프 크리프 금지) — 작업 중 발견한 별개의 개선 아이디어는 구현하지 말고 보고에 남긴다.
- 지시서가 여러 개 연달아 오면(추가 지시가 새 메시지로 도착) 먼저 받은 것부터 끝내고, 사용자가 명시적으로 순서를 바꾸라고 하지 않는 한 순서를 지킨다.

## 완료 판정

- 지시서의 "완료 기준" 각 항목이 실행 검증 문서에서 O/X로 표시돼 있다.
- 불변 제약 위반이 없음을 `rg` 등으로 직접 확인했다.
- 검증 문서와 함께 커밋되어 push까지 끝났다.
