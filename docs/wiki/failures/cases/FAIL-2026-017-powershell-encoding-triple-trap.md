# FAIL-2026-017 — PowerShell 5.1 인코딩 3중 함정: **독립 검수자에게 깨진 입력을 줬다**

- 상태: 해결 (`skills/common/powershell-encoding.md` 신설 · `run-codex-review.ps1` 수정)
- 최초 발생일: 2026-07-11 이전 (`run-executor.ps1` BOM 문제 — 소급 확인)
- 최근 발생일: 2026-07-14
- 관련 DI: P0-06 · 05H-R2 검수 · 코덱스 독립 검수 배선
- 구성요소: executor-orchestration
- failureClass: config_side_effect
- 심각도: 높음 (**계측기·검수자에게 오염된 입력이 들어가면 판정 자체가 오염된다**)

## 발생 상황

한국어가 섞인 텍스트를 PowerShell 5.1로 다루는 모든 경로에서 인코딩이 깨진다.
**세 계층이 각각 다른 기본 인코딩을 쓰기 때문이다.**

```
PowerShell 5.1 기본 인코딩  = 시스템 코드페이지 (한국어 Windows = 949)
.NET / 파일 실체            = UTF-8
cmd 파이프                  = 콘솔 코드페이지
```

**한 계층만 맞춰도 다른 계층에서 깨진다.** 그래서 "UTF-8로 썼다"가 증거가 되지 못한다.

## 관찰된 증상과 영향

| # | 경로 | 결과 |
| --- | --- | --- |
| 1 | **스크립트 파일에 BOM이 없다** | PS 5.1이 **스크립트 소스의 한글 리터럴**을 코드페이지로 읽어 파괴한다. `run-executor.ps1`의 `'^##\s+허용 파일'` 정규식이 매치에 실패 → `FILE-CLAIMS.paths`가 **항상 0** → **P0-06 사전 차단이 한 번도 작동한 적이 없다**. 게이트는 `claimConflictCount: 0`(평화)을 보고했다 — **빈 배열끼리의 비교였다** |
| 2 | **`Get-Content -Raw` (인코딩 미지정)** | **읽는 순간** UTF-8 한글이 파괴된다. `state에 같은 id 2회` → `state??媛숈? id 2??` |
| 3 | **`Set-Content -Encoding UTF8`** | **BOM을 붙인다** (`EF BB BF`) |
| 4 | **`type file \| prog`** | cmd 파이프가 **코드페이지로 재인코딩**한다 |
| 5 | **명령행 인자로 긴 한글 전달** | 인자도 콘솔 인코딩을 탄다 |
| 6 | **git 출력 캡처** | `[Console]::OutputEncoding`을 UTF-8로 안 바꾸면 한글 diff가 깨진다 |

### ★ 가장 위험했던 것 — **독립 검수자를 오염시켰다** (2026-07-13)

검수자가 `run-codex-review` 입력 패킷을 만들 때 **경로 2·1에 걸렸다.**
→ 코덱스에게 **깨진 fixture**가 들어갔다.
→ 코덱스는 **정상 fixture 5종을 "malformed JSON"으로 오독**하고 **없는 결함을 보고했다.**
→ **하마터면 그 보고로 실행자를 반려할 뻔했다.**

> **독립 검수자에게 준 입력이 오염되면 그가 내는 판정도 오염된다. 검수 배선도 검수 대상이다.**

**U+FFFD(대체 문자)가 0이라 자동 검사도 못 잡았다** — 깨진 바이트들이 **유효한 UTF-8 시퀀스**로 재해석됐기 때문이다.

## 발생 이유

- **직접 원인**: PS 5.1의 기본 인코딩이 시스템 코드페이지(949)다.
- **근본 원인**: **계층마다 기본값이 다른데, 어느 계층에서 깨졌는지 확인하는 절차가 없었다.**
- **기여 원인 (★ 진단 과정의 오류)**:
  1. **1차 오진 — BOM이 범인이다.** `Set-Content -Encoding UTF8`이 BOM을 붙이는 것을 확인하고 고쳤다. **왕복은 계속 깨졌다.**
  2. **2차 오진 — cmd 파이프가 범인이다.** `<` 리다이렉션으로 바꿨다. **여전히 깨졌다.**
  3. **실체 — `Get-Content -Raw`가 읽는 순간 파괴한다.**

  > **BOM은 실재했지만 범인이 아니었다.** **"거기 있다"와 "그것이 원인이다"는 다르다.**
  > CLAUDE.md의 **"프록시로 단정하지 마라"**를 검수자가 스스로 어겼다. **두 번 헛짚고서야 왕복 시험으로 실체에 닿았다.**

## 검토한 해결 방법

1. 시스템 코드페이지를 UTF-8로 바꾼다 (× — 다른 도구를 깨뜨린다. 환경 의존)
2. PowerShell 7로 이전한다 (△ — 범위가 크다. 별도 과제)
3. **명시적 인코딩 규칙 + 왕복 시험 강제** (○)

## 선택한 해결 방법

**규칙을 스킬로 승격하고, 배선에 자동 검사를 넣는다.**

| 하는 일 | **쓸 것** | **쓰지 말 것** |
| --- | --- | --- |
| 스크립트 파일 저장 | **BOM 있는 UTF-8** | BOM 없는 UTF-8 ← 한글 리터럴이 깨진다 |
| 파일 읽기 | `[IO.File]::ReadAllText($p, UTF8)` | `Get-Content $p -Raw` (인코딩 미지정) |
| 파일 쓰기 | `[IO.File]::WriteAllText($p, $t, UTF8Encoding($false))` | `Set-Content -Encoding UTF8` (BOM) |
| 프로세스 전달 | 파일 + **stdin 리다이렉션** `cmd /c "prog - < file"` | `type file \| prog` · 긴 한글 인자 |
| git 출력 캡처 | `[Console]::OutputEncoding = UTF8` 먼저 | 그냥 캡처 |

**자동 검사** (`run-codex-review.ps1`에 반영):

```powershell
if ($b[0..2] -eq 0xEF,0xBB,0xBF) { throw "입력 패킷에 BOM" }
if ($text.Contains([char]0xFFFD)) { throw "깨진 문자(U+FFFD)" }
```

**★ 그러나 U+FFFD 검사는 불충분하다** (이번 사고에서 0이었다).
**정본은 왕복 시험이다** — 한글을 써서 → 넘겨서 → 돌려받아서 → **원본과 비교하라.**

## 판단 기준

**"UTF-8로 썼다"는 주장은 증거가 아니다. 계층을 통과한 뒤에 재라.**

## 검증 결과

- `run-codex-review.ps1`에 BOM 추가 → **코덱스가 한글을 정확히 그대로 돌려줬다**(왕복 성공)
- `run-executor.ps1`은 **원래 BOM이 있다**(`efbbbf`) — 그래서 그 한글 정규식이 작동한다. **검수자 스크립트만 없었다**
- 저장소 `.ps1` 전수 조사: **인코딩 미지정 `Get-Content` 없음** (`run-executor.ps1:24`는 이미 `-Encoding UTF8`)
  - **정정**: 전수 조사 정규식이 `Get-ContentText`(사용자 정의 함수)를 `Get-Content`로 **오탐**했다.
    하마터면 없는 결함을 또 보고할 뻔했다. **정규식 매치는 증거가 아니다.**

## 재발 방지

1. **`skills/common/powershell-encoding.md`를 읽어라** — PowerShell로 한글을 다루는 모든 작업에서.
2. **새 `.ps1`은 BOM 있는 UTF-8로 저장한다.**
3. **독립 검수자/하네스에 넘기는 입력 패킷은 왕복 시험으로 검증한다.**
4. **증상을 고치지 말고 원인을 지목하라.** BOM을 고쳤는데 여전히 깨지면 **다른 곳이 범인이다.**

## 후속 작업과 잔여 위험

- `run-executor.ps1`의 `Set-Content -Encoding UTF8`(BOM) 7건 — .NET 독자가 BOM을 벗기므로 **현재는 무해**. 다만 규칙 통일 필요.
- **잔여 위험**: 이 함정은 **환경 의존**이다. 한국어 Windows(cp949)가 아닌 환경에서는 다르게 나타난다.
  **PowerShell 7 이전**이 근본 해결이지만 별도 과제다.
