# ADR-004 — 발사 ACK 규약은 결함이다: 검출을 stdout이 아니라 파일로 옮긴다

- 상태: **SUPERSEDED by ADR-010 (2026-07-12)** — 이전 상태: 제안(확정된 적 없음).
  **문제 진단("ACK 규약은 결함이다")은 계승한다. 그러나 여기서 고른 해법(파일 기반 ACK)은 폐기한다** —
  실행자가 파일을 써야 하므로 **여전히 모델의 순종에 의존**하기 때문이다.
  대체 해법: `--replay-user-messages` 기반 **Transport Receipt**(`ADR-010`). **말이 아니라 바이트로 판정한다.**
  *이 줄은 덮어쓰기가 아니라 추가다 — 기록은 이력이지 현재 상태가 아니다.*
- 일시: 2026-07-11
- 제안: 검수자 세션(Claude)
- 근거 문서: FAIL-2026-013 · `skills/common/executor-launch.md`

## 1. 상황
FAIL-2026-013(프롬프트가 인자 경계에서 잘려 **지시가 도착하지 않음**)의 재발 방지로 **"맨 첫 줄에 `ACK-<taskId>`를 출력하라"**를 발사 프롬프트에 넣고, 없으면 산출물 폐기하기로 했다.

**실측: ORCH-01·ACTOR-01·FIX-04·FIX-05·FIX-06 전부 ACK를 출력하지 않았다(5/5 실패).** 그럼에도 다섯 건 모두 지시서 allowlist를 정확히 지켰다 — **지시는 도착했다.**

원인(실체): `claude -p`는 stdout에 **최종 메시지만** 내보낸다. 모델이 수행요약부터 쓰면 "첫 줄 ACK"는 존재할 자리가 없다.

## 2. 선택지
(A) 규칙대로 ACK 없는 산출물을 전부 폐기 — **정상 산출물 5건을 버리게 된다.** 규칙이 현실과 어긋난다.
(B) ACK 요구를 **마지막 줄**로 옮긴다 — 여전히 stdout 의존, 모델 재량.
(C) **ACK를 파일로 받는다** — 실행자가 시작 시 `outputs/ack-<taskId>.txt`를 쓰게 하고, `launch-check` 하네스가 **파일 존재**로 지시 도착을 판정한다. stdout은 휘발하지만 파일은 남는다.

## 3. 선택
**(C).** 같은 이유로 QUOTA-POLICY의 "죽기 전 유언 3줄"도 stdout이 아니라 **verification 문서(`상태: 미완(한도)`)** 를 1차 근거로 삼는다.

## 4. 판단 기준
**말로 강제되지 않는 것은 검사로 내린다.** 프롬프트 지시는 5/5 무시됐다 — 이것이 실증이다.

## 5. 결과
- 코덱스 H-00(`launch-check`)을 파일 기반 ACK로 수정한다. **현행 stdout 기반 그대로 쓰면 정상 산출물을 전부 오탐 처리한다.**
- 발사 프롬프트 템플릿을 "시작하자마자 `outputs/ack-<taskId>.txt` 에 taskId를 써라"로 바꾼다.

## 6. 되돌림 조건
파일 기반 ACK도 무시되면(모델이 파일을 안 씀) 발사 방식 자체(프롬프트 전달)를 재설계한다 — ORCH-03(자식 프로세스 핸들 소유).
