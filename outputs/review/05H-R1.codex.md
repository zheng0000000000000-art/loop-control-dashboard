### 지표는 통과하지만 목적을 배신하는 경로

1. fixture-f는 pending 없이 실행되므로 exit 1이 난다.
2. CLI pending은 별도에서 금지된다고 주장되어 완료 기준을 통과할 수 있다. [DIRECTIVE.md:123](C:/Users/1/AppData/Local/Temp/codex-rev-test/DIRECTIVE.md:123)
3. 내부 checker 완료 기준은 “state X 1개 + log X 없음 + Pending=X PASS”만 검증한다. [DIRECTIVE.md:124](C:/Users/1/AppData/Local/Temp/codex-rev-test/DIRECTIVE.md:124)
4. 하지만 실제 구현은 “log X 없음”을 검사하지 않는다. 실패 로그 X가 있어도 pending이면 PASS다.
5. 따라서 모든 기존 fixture와 gate가 초록이어도, 실패로 기록된 전이를 pending으로 지정한 in-process 호출은 목적 기준을 배신한다.

### fixture가 못 잡는 것

fixture-f는 [CHANGES.diff:33-34](C:/Users/1/AppData/Local/Temp/codex-rev-test/CHANGES.diff:33)의 실패-only 로그와 [CHANGES.diff:48-50](C:/Users/1/AppData/Local/Temp/codex-rev-test/CHANGES.diff:48)의 state 적용을 잘 만든다. 그러나 pending 인자를 같은 ID로 주는 변종이 없다.

못 잡는 구체 입력은 위 `PENDING-FAILED`와 동일하며, fixture-f 기준으로는 `PendingTransitionId="FAILED-BUT-APPLIED"`를 추가한 호출이다.

read-only 샌드박스라 `dotnet build`, fixture 실행, `dotnet run`은 수행하지 않았다. 판단은 `DIRECTIVE.md`, `CHANGES.diff`, `HandoffIntegrityChecker.cs` 정적 대조 기준이다.
