## 배경 — 지금 `launch-check`는 죽은 게이트다

```csharp
// server/Harness/LaunchCheckCli.cs:36-68
var expectedAck = $"ACK-{taskId}";

return exists && ackFound && !quotaSignal ? 0 : 1;
```

**실측(5/5)**: ORCH-01·ACTOR-01·FIX-04·FIX-05·FIX-06 **전부 stdout에 ACK를 내지 않았다.** 그런데 **5건 다 지시서를 정확히 지켰다 — 지시는 도착했다.** 즉 이 하네스는 **정상 산출물을 오탐으로 죽인다.** 그래서 **아무 발사 경로에도 배선되지 않았다**(Registry 등록만 있고 자동 호출 0건). **비활성·비권위 게이트다.**

ADR-004는 대안으로 **파일 기반 ACK**를 골랐는데, **그것도 여전히 모델의 순종에 의존한다.** 실행자가 파일을 안 쓰면 끝이다.

## 이번에 바꾸는 것 — **말이 아니라 바이트로 판정한다**

검수자가 2026-07-12에 **실측으로 증명했다**: 설치된 Claude CLI가 `--replay-user-messages`를 지원한다.

```
claude -p --verbose --input-format stream-json --output-format stream-json --replay-user-messages
```

stdin으로 보낸 user 메시지를 **CLI가 그대로 되돌려준다.** 그 본문의 sha256이 보낸 것과 같으면 **전송 무결성이 증명된다. 모델은 관여하지 않는다.**

**첫 실행에서 즉시 값어치를 증명했다**: 프롬프트가 `ACK�� ����ض�`로 깨져 도착한 것을 잡았다(PowerShell 5.1이 stdin에 UTF-8을 안 쓴다). **receipt가 없었으면 "모델이 이상하다"고 오진했을 것이다.**

### ★ receipt의 권위 범위 — 넘겨짚지 마라

**replay hash는 "내가 보낸 바이트 = CLI가 받은 바이트"만 증명한다.**
**모델의 이해·주의·순종은 증명하지 않는다.** 판정 문자열에 `MODEL_RECEIVED`·`MODEL_UNDERSTOOD`·`ACK_VALID` 같은 표현을 **쓰지 마라.**

## 할 일

### 1. `launch-check`를 transport evidence 기반으로 교체

- **입력**: taskId + transport evidence 파일 경로(발사 래퍼가 만든다. 아직 없다 — 아래 계약을 네가 먼저 고정한다).
- **판정**:
  - `payloadSha256 == replaySha256` → **PASS(exit 0)**
  - 불일치 → **`TRANSPORT_INVALID`(exit 1)**
  - evidence 파일 없음 / replay 이벤트 없음 / 2건 이상 / JSON 파싱 실패 / CLI가 옵션 미지원 → **전부 fail-closed(exit 1)**
- **ACK 관련 코드는 전부 제거한다.** 하위 호환 남기지 마라. **명령행 프롬프트 방식으로 조용히 폴백하는 경로를 검출하면 FAIL이다.**

### 2. transport evidence 계약을 고정한다 (`outputs/launch/<TaskId>.transport.json`)

```json
{
  "schemaVersion": 1,
  "taskId": "LEDGER-05",
  "cliVersion": "...",
  "sourceSha256": "<64자리 전체>",
  "payloadSha256": "<64자리 전체>",
  "replaySha256":  "<64자리 전체>",
  "sourceByteLength": 0,
  "payloadByteLength": 0,
  "replayByteLength": 0,
  "replayEventCount": 1,
  "pid": 0,
  "startedAt": "...",
  "exitedAt": "...",
  "verdict": "TRANSPORT_VALID | TRANSPORT_INVALID"
}
```

- **sha256은 64자리 전체를 기록한다.** 앞 12자리 같은 축약 금지.
- **프롬프트 본문 자체를 evidence나 usage ledger에 복제하지 마라.** 해시와 길이만 남긴다.

### 3. Context Budget 검사 (v9 §, 사람 승인 필요한 정책값은 네가 정하지 마라)

**v9는 명시한다: "Budget 초과 자체는 실패가 아니다."** 그러니 **단순 상한 초과로 FAIL시키지 마라.** 규칙은 이렇다:

```
payloadByteLength > policy.maxPayloadBytes
  AND splitDecision 없음
  AND budgetExceptionReason 없음
      → FAIL
```

- 정책 파일 경로만 읽어라: `docs/handoff/LAUNCH-BUDGET.json`. **없으면 budget 검사는 `skipped`로 두고 exit code에 반영하지 마라**(정책값은 사람이 실측 후 승인한다. **네가 숫자를 지어내지 마라**).
- **`maxRequiredSummaryChars=16000`을 바이트 상한으로 전용하지 마라.** 한글은 문자 수 ≠ UTF-8 바이트 수다.

## ★ Test-first — **현재 실행이 FAIL하는 것을 먼저 증명해라**

지금은 transport evidence 파일이 **존재하지 않는다.** 새 `launch-check`는 **현재 상태에서 반드시 exit 1**이어야 한다. **그것부터 보여라.** (그 다음 실행자가 래퍼를 고쳐 evidence를 만들면 PASS로 바뀐다.)

## 필수 반증 시험 (사본에서. 원본 무접촉)

1. **인코딩 손상 fixture**: PowerShell 기본 stdin writer로 한글을 보낸 payload → **mismatch로 잡힌다**
2. 같은 fixture를 **UTF-8 바이트 직접 쓰기**로 보내면 → **match**
3. **다중 언어 fixture 통과**: 한글 · emoji · 따옴표 · 역슬래시 · 여러 줄 · CRLF/LF
4. replay 이벤트가 **없으면** 실패
5. replay 이벤트가 **둘 이상이면** 실패
6. replay JSON을 **변조하면** 실패
7. 프롬프트 **한 글자만 바꿔도** 실패
8. **긴 프롬프트**가 명령행 길이 제한과 무관하게 전달된다

**통과를 믿기 전에 실패시킬 수 있음을 먼저 증명해라.**
