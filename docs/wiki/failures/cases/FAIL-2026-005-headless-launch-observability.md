# FAIL-2026-005 — 헤드리스 실행자 미실행을 진행 중으로 오판

- 상태: 해결됨
- 최초 발생일: 2026-07-10
- 최근 발생일: 2026-07-10
- 관련 DI: 향후 #13 실행자 사다리 관측 요구
- 구성요소: executor-orchestration
- failureClass: observability, known_failure
- 심각도: medium

## 발생 상황

리팩토링 실행자를 `Start-Process powershell -Command "Get-Content prompt | claude -p ... *> log"` 형태로 발사했다. 발사 명령은 `launched`를 반환했다.

## 관찰된 증상과 영향

로그 파일은 0바이트로 남았고 산출물인 `CliRouter.cs`도 생기지 않았다. 검수자는 이를 무거운 작업이 진행 중인 상태로 오판해 기다렸지만, 실제로는 실행자 프로세스가 붙지 않아 아무 작업도 일어나지 않았다. 이로 인해 수 분을 낭비했다.

## 발생 이유(직접·근본·기여, 미확정은 가설 표시)

- 직접 원인: `Get-Content | claude` stdin 파이프가 `Start-Process` 래퍼 컨텍스트에서 전달되지 않아 실행자가 입력 없이 종료했다.
- 근본 원인: `launched` 반환을 실행 성공으로 간주했고, 실행 여부를 PID나 산출물로 확인하지 않았다.
- 기여 요인: 로그 리다이렉트(`*>`)가 해당 래퍼에서 기대대로 관측을 남기지 않아 로그 공백이 생겼다.
- 미확정: `claude -p` 출력 버퍼링이 완료 전 로그 가시성에 얼마나 영향을 줬는지는 별도 계측하지 않았다.

## 검토한 해결 방법

- 프롬프트를 stdin 대신 인자로 직접 전달한다.
- `RedirectStandardOutput`과 `-PassThru`로 프로세스와 로그를 확실히 확보한다.
- 발사 직후 PID, `HasExited`, 산출물 존재 여부로 실행 여부를 확인한다.

## 선택한 해결 방법

프롬프트를 인자로 직접 전달하고, `RedirectStandardOutput`을 사용하며, 발사 직후 `HasExited`와 산출물 존재를 확인하는 방식으로 재발사했다. 재발사(`rec6b`)에서 정상 실행을 확인했다.

## 판단 기준

명령이 반환됐다는 사실과 실행자가 실제 작업을 수행했다는 사실은 다르다. 실행 성공은 PID, 로그, 산출물 같은 관측으로 확인해야 한다.

## 검증 결과

초기 발사는 로그 0바이트와 산출물 부재로 미실행 상태였음이 확인됐다. 수정된 발사 방식에서는 재발사(`rec6b`) 후 정상 실행이 확인됐다.

## 재발 방지

- 향후 #13 실행자 사다리에 실행 PID 확인과 로그 보장을 필수 요구로 포함한다.
- 헤드리스 실행자는 발사 직후 `HasExited`와 산출물 존재를 확인한다.
- 완료 판정은 "launched" 문자열이 아니라 산출물 존재와 검수 결과를 기준으로 한다.

## 후속 작업과 잔여 위험

- `claude -p` 출력 버퍼링 방식에 따라 완료 전 로그가 비어 보일 수 있다.
- 로그가 비어 보여도 산출물 기준 판정을 우선한다.
- 장기적으로는 실행자별 표준 래퍼가 필요하다.

## 발생 이력

- 2026-07-10: 로그 0바이트와 `CliRouter.cs` 부재 확인.
- 2026-07-10: 인자 직접 전달과 출력 리다이렉션 방식으로 재발사(`rec6b`) 후 정상 실행 확인.
