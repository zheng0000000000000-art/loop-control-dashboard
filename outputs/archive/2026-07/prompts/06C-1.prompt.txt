너는 06C-1 실행자다. 저장소 루트는 C:\Users\1\Documents\Local-First Workflow Dashboard 다.
**현재 브랜치는 `wp/state-integrity`다(통합 branch). 브랜치를 바꾸지 마라.**

## ★★ 먼저 읽어라 — 이 함정에 전임자가 빠졌다

**지금 이 저장소의 canonical 상태는 오염돼 있다. 그리고 그건 고쳐진 게 아니라 "탐지된" 것이다.**

```
docs/handoff/WORKSTATE.applier-log.jsonl:20
  {"transitionId":"DI0004-BLOCKED-CODEX","result":"handoff-integrity-failed exit=1","exitCode":1,...}
docs/handoff/WORKSTATE.json:384
  같은 id가 appliedTransitions에 들어 있다
```

**적용이 실패했다고 자기 로그가 말하는 전이가 상태에 남아 있다.** (= 네가 지금 고치려는 결함 2, rollback 없음의 흔적)

그래서:

```
dotnet run --project server -c Release -- handoff-integrity      → exit 1
   failures: state-transition-not-logged (DI0004-BLOCKED-CODEX)  ← 정확히 1건
```

**이것은 버그가 아니다. 설계된 참 양성이다.** 05H-R1이 방금 이 탐지를 살려냈다.

### 그래서 네게 무슨 일이 생기는가

네가 만들 §1 순서는 **reconciliation이 먼저**다:

```
3. reconciliation (내부 checker, pending 없음)
     FAIL → 일반 전이 거부 exit 1 state-corrupted-preapply
```

**즉 06C-1이 완성되면 canonical WORKSTATE에 대한 NORMAL 전이는 전부 거부된다. 상태 기계가 얼어붙는다.**
**그것이 의도다.** 상태를 믿을 수 없으면 상태를 바꿔선 안 된다. 해동은 **06C-2 trust-origin 부트스트랩 + 사람 결재**가 한다(별도 지시서, 같은 land gate).

### ★ 그러므로 — 절대 하지 마라

**"실제 저장소에서 NORMAL 전이가 되게 만들려고" reconciliation을 완화하거나, 예외·플래그·allowlist·bypass를 넣지 마라.**

**전임자(05H)가 정확히 그것 때문에 반려됐다.** 프롬프트가 "at-rest → exit 0"을 요구했고, 규칙을 지키면 불가능했고,
그래서 규칙을 완화했다. (그 프롬프트는 검수자가 잘못 쓴 것이다. 이번엔 그런 모순을 두지 않았다.)

**요구가 서로 모순돼 보이면 완화하지 말고 보고해라. 보고는 감점이 아니다. 완화는 반려다.**

## 완료 기준 2·3·5·6·7·8은 fixture에서 실증하라 — canonical에 적용하지 마라

지시서 §8의 왕복 시험(NORMAL prepare→apply → exit 0 등)은 **실제 `docs/handoff/WORKSTATE.json`에 대해서는 성립하지 않는다**(위 이유).

```
canonical WORKSTATE·applier-log 수정 금지. state-transition을 canonical에 apply하지 마라. (사람 게이트이기도 하다)
```

**대신 저장소 사본(temp copy)에서 시험하라:**

1. `$env:TEMP\06c1-<시험명>\` 에 저장소를 복사한다.
2. 사본의 `docs/handoff/WORKSTATE.json` + `WORKSTATE.applier-log.jsonl`을 **known-good 쌍**으로 교체한다.
   → `docs/qa/fixtures/reconciliation/fixture-c/` 가 known-good 불변 스냅샷이다(reconciliation exit 0).
3. 그 사본에서 prepare→apply 왕복을 돌린다.
4. **시험마다 사본을 새로 만들어라.** 앞 시험이 사본 상태를 바꾼다(검수자 실측 함정).

`--root` 같은 production 경로 플래그를 새로 만들지 마라. 그건 canonical 우회 통로가 된다.
`--dry-run`은 이미 있다(`StateApplierCli.cs:426`).

## 선행 의존물 — 이미 있다 (실측 확인)

`server/Harness/HandoffIntegrityChecker.cs`가 제공한다:

- `ReconciliationOptions(WorkstatePath, ApplierLogPath, PendingTransitionId)` — in-process 호출용
- `lookupSuccess(transitionId)` → `{ exists, schemaVersion, transitionContractSha256? }` — 네 idempotency 판정용
- **`HarnessRegistry`에 등재돼 있지 않다.** 그대로 둬라 — CLI에 노출되면 pending 면제가 우회 통로가 된다.

**⚠ 아직 아무도 실증하지 못한 것**: 내부 checker의 **pending 면제 PASS 경로**(state에 X 1회 + log에 X 없음 + Pending=X → PASS).
CLI 미노출이라 외부에서 못 돌린다. **네가 §4-7의 적용후 검사에서 `PendingTransitionId=id`로 호출하는 그 순간이 첫 실증이다.**
**그 결과를 보고서에 명시적으로 적어라.** 검수자가 그것만 기다리고 있다.

## 한 줄 요약

**결함 4개를 고친다.** ①멱등이 reconciliation보다 먼저(`StateApplierCli.cs:84`) ②rollback 없음 ③`BuildCandidate`의 `UtcNow` 비결정성 ④id만 같으면 멱등 성공(내용 결속 없음).
**핵심은 순서다** — reconciliation → existing-transition → (미적용 시) pre-state hash.

## 절대 규칙

- **`docs/handoff/WORKSTATE.json`·`WORKSTATE.applier-log.jsonl` 수정 금지.** append-only 증거다. **`DI0004-BLOCKED-CODEX`를 지우지 마라.**
- **`server/Harness/HandoffIntegrityCli.cs`·`HandoffIntegrityChecker.cs` 무접촉** — 05H 영역. **읽기만.**
- **`server/Harness/DiCompletionCheckCli.cs`·`ClaimCheckCli.cs` 무접촉** — `CODEX-GATE-04` 영역.
- **`--human-decision <임의파일>` 폐기.** PHASE_CHANGE/RECOVERY/REPLAY는 이번 WP에서 **항상 exit 1 `trusted-human-receipt-required`**(receipt ledger 부재). **골격만 남기고 활성화하지 마라.**
  (이 구멍으로 검수자가 실제로 자기 승인을 위조한 적이 있다. 그래서 fail-closed다.)
- **`CLAIMED_ACTOR`와 `VERIFIED_HUMAN_APPROVAL`을 섞지 마라.**
- **실패 주입은 결정적 test seam으로.** production에 노출되는 플래그 금지.
- **git commit/push 금지. 브랜치 변경 금지. 결재·반입·발사 금지.**
- **`measure`는 `dotnet run --project server -c Release -- measure dev-pack`으로.** exe 직접 호출은 부모 폴더를 봐서 exit 2가 난다(실측).
- 함수마다 한국어 기능 주석 1줄. 빌드는 `-c Release`.
- **⚠️ `di-completion-check`는 하위 검사를 Debug 바이너리로 돌린다**(실측 결함, `CODEX-GATE-04`가 고친다). 코드를 고쳤으면 **`dotnet build server`(Debug)도 한 번** 돌려야 게이트가 새 코드를 본다.

## 판정선 (지시서 §8) — 2·3·5·6·7·8은 **사본+known-good fixture에서**

```
 1. dotnet build server -c Release                              → 0
 2. NORMAL prepare→apply 왕복 (사본)                            → 0            ★결함1
 3. 같은 envelope 재적용 (사본, 최초 성공 후)                    → 0 idempotent  ★재시도 생존
 4. 손 위조: state에 가짜 id만, log엔 없음 → 전이               → 1 state-corrupted-preapply  ★
 5. 같은 id·다른 request(v2)                                    → 1 transition-id-collision   ★결함4
 6. 기존 id가 v1 로그                                           → 1 legacy-idempotency-unverifiable
 7. prepare 후 candidate 1바이트 변조 → apply                    → 거부(재계산 bytes로만 write) ★TOCTOU
 8. test seam 적용후 실패 → 복원                                → 1 ROLLED_BACK, hash==preimage ★결함2
 9. FATAL 4분기 각각 도달
10. transitionKind ∈ {PHASE_CHANGE,RECOVERY,REPLAY}             → 1 trusted-human-receipt-required
11. state-transition-callsite-check                            → legacyCallsiteCount 0
12. measure dev-pack (-c Release)                              → 0
13. ★ canonical 저장소에서 handoff-integrity                    → **exit 1** (DI0004-BLOCKED-CODEX 1건)
       이게 정상이다. 0으로 만들려 하지 마라.
```

**통과를 믿기 전에 실패시킬 수 있음을 먼저 증명해라.**

## 목적 기준 (지표로 우회할 수 있다 — ADR-005)

**"상태를 믿을 수 없으면 상태를 바꿀 수 없다."**

우회로: reconciliation을 apply 경로에서 빼거나, 조건부로 건너뛰거나, canonical에만 예외를 주는 것.
**전부 목적 미달이다. 자진 신고 없이 하면 반려다. 신고하면 감점이 아니다.**
